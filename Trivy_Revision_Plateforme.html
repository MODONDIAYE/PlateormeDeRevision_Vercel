1 - Securite dans le DevOps

0/5

Le defi de la securite en DevOps : Les pipelines CI/CD deployent du code rapidement — parfois plusieurs fois par jour. Cette vitesse est un avantage pour le business, mais un risque pour la securite. Si une image Docker contient une vulnerabilite connue (CVE), elle peut etre deployee en production en quelques minutes. Sans scan automatise, personne ne le detecte avant qu'un attaquant l'exploite.

Shift-Left Security : Le concept "Shift-Left" consiste a deplacer les controles de securite le plus tot possible dans le cycle de developpement. Au lieu de scanner en production (trop tard), on scanne au moment du build, du commit, voire dans l'IDE du developpeur. Plus une faille est detectee tot, moins elle coute cher a corriger.

Attaques de la supply chain : Les applications modernes dependent de centaines de librairies open source (npm, pip, Maven). Un attaquant peut compromettre une seule dependance et impacter des milliers de projets (exemples : log4shell, event-stream, SolarWinds). Scanner les dependances est devenu critique.

CVE et vulnerabilites : Un CVE (Common Vulnerabilities and Exposures) est un identifiant unique pour une faille de securite connue (ex: CVE-2021-44228 = Log4Shell). Les bases de donnees NVD (National Vulnerability Database) et GHSA (GitHub Security Advisories) repertorient ces failles avec leur severite (CVSS score).

Securite des conteneurs : Les images Docker contiennent un OS de base (Alpine, Debian, Ubuntu) avec ses propres paquets et vulnerabilites. Meme si votre code est parfait, l'image de base peut contenir des dizaines de CVE critiques. Scanner les images avant deploiement est indispensable.

  • Shift-Left = securite le plus tot possible dans le pipeline.
  • Supply chain = dependances compromises = risque massif.
  • CVE = identifiant unique d'une vulnerabilite connue.
  • NVD + GHSA = bases de donnees de vulnerabilites.
  • Images Docker = OS de base + paquets = surface d'attaque.
  • Sans scan automatise, les failles arrivent en production.

2 - Presentation de Trivy

0/5

Qu'est-ce que Trivy ? Trivy est un scanner de securite open source "all-in-one" developpe par Aqua Security. Il detecte les vulnerabilites, les misconfigurations, les secrets exposes et les problemes de licences dans les images Docker, les systemes de fichiers, les depots Git et les clusters Kubernetes.

Aqua Security : Aqua Security est l'entreprise israelienne qui developpe Trivy. Fonde en 2015, Aqua est un leader de la securite cloud-native. Trivy a ete open-source des le depart et est devenu le scanner le plus populaire de l'ecosysteme cloud-native (20k+ stars GitHub).

Scanner all-in-one : Contrairement a d'autres outils specialises (Clair pour les images, TFSec pour Terraform), Trivy fait tout : vulnerabilites OS, vulnerabilites applicatives, secrets, IaC misconfigurations, licences. Un seul outil pour tout scanner.

Cibles de scan : Trivy peut scanner : des images conteneurs (Docker, OCI), des systemes de fichiers locaux, des depots Git distants, des clusters Kubernetes, des fichiers IaC (Terraform, CloudFormation, Dockerfile, Kubernetes YAML). C'est l'outil le plus polyvalent.

Comparaison avec les alternatives : Clair (RedHat) = images uniquement, plus complexe a deployer. Snyk = commercial, excellent mais payant. Grype (Anchore) = similaire a Trivy mais moins de fonctionnalites. Trivy se distingue par sa simplicite, sa rapidite et sa couverture complete.

  • Trivy = scanner all-in-one par Aqua Security (Israel, 2019).
  • Open source, 20k+ stars GitHub, le plus populaire.
  • Cibles : images, filesystem, git repos, K8s, IaC.
  • Detecte : vulnerabilites, secrets, misconfigs, licences.
  • Simple a utiliser : un seul binaire, zero configuration.
  • Alternatives : Clair, Snyk, Grype — Trivy est le plus complet.

3 - Scan d'Images Docker

0/5

La commande trivy image : C'est la commande de base pour scanner une image Docker. Syntaxe : trivy image nginx:latest. Trivy telecharge sa base de donnees de vulnerabilites, analyse chaque couche de l'image, et affiche un rapport avec les CVE trouvees, leur severite et les versions corrigees.

Niveaux de severite : Trivy classe les vulnerabilites en 4 niveaux : CRITICAL (score CVSS 9.0-10.0, exploitation facile, impact maximal), HIGH (7.0-8.9), MEDIUM (4.0-6.9), LOW (0.1-3.9). On peut filtrer avec --severity CRITICAL,HIGH pour ne voir que les plus graves.

Filtrage et ignore : Le fichier .trivyignore permet d'ignorer certains CVE (faux positifs, vulnerabilites acceptees). Syntaxe : un CVE par ligne (ex: CVE-2021-44228). On peut aussi filtrer par type avec --vuln-type os,library pour scanner uniquement les paquets OS ou applicatifs.

Images de base et strategie : Les images basees sur Debian/Ubuntu contiennent souvent 100+ vulnerabilites. Strategies de reduction : utiliser Alpine (minimaliste, musl libc), utiliser des images distroless (Google, sans shell ni package manager), ou builder en multi-stage pour minimiser la surface d'attaque.

Scan de registres distants : Trivy peut scanner des images directement depuis un registre sans les pull localement : trivy image --remote registry.example.com/app:v1. Il supporte Docker Hub, ECR, GCR, ACR et tout registre compatible OCI.

  • trivy image <nom> = commande de base pour scanner.
  • 4 niveaux : CRITICAL, HIGH, MEDIUM, LOW.
  • --severity CRITICAL,HIGH = filtrer les plus graves.
  • .trivyignore = fichier pour ignorer des CVE specifiques.
  • Alpine/Distroless = reduire la surface d'attaque.
  • Scan distant possible sans pull local.

4 - Vulnerabilites & Secrets

0/5

Base de donnees CVE : Trivy maintient sa propre base de donnees de vulnerabilites, mise a jour quotidiennement. Elle agregre les donnees de : NVD (National Vulnerability Database, USA), GHSA (GitHub Security Advisories), et les advisories specifiques des distributions (Alpine, Debian, RHEL, Ubuntu). Au premier scan, Trivy telecharge cette DB (~30 Mo).

Paquets OS vs paquets applicatifs : Trivy distingue deux categories : les paquets OS (apt, apk, rpm — vulnerabilites de l'image de base) et les paquets applicatifs (npm, pip, gem, Maven, Go modules — vulnerabilites de vos dependances). Les deux sont scannes automatiquement.

Detection de secrets : Trivy detecte les secrets exposes dans le code et les images : cles API, tokens, mots de passe, cles SSH, certificats. Il utilise des patterns regex et des heuristiques pour identifier les secrets. Commande : trivy fs --scanners secret .

Scan de licences : Trivy peut identifier les licences des dependances (MIT, GPL, Apache, etc.) et alerter si une licence restrictive (GPL, AGPL) est utilisee dans un projet commercial. Utile pour la conformite legale.

SBOM (Software Bill of Materials) : Trivy peut generer un SBOM — un inventaire complet de tous les composants logiciels dans une image ou un projet. Formats supportes : CycloneDX, SPDX. Le SBOM est de plus en plus exige par les reglementations (Executive Order US, NIS2 EU).

  • DB Trivy = NVD + GHSA + advisories distributions, maj quotidienne.
  • Paquets OS (apt/apk/rpm) + applicatifs (npm/pip/gem/Maven).
  • Detection de secrets : cles API, tokens, mots de passe.
  • Scan de licences pour conformite legale (GPL, MIT, Apache).
  • SBOM = inventaire complet (CycloneDX, SPDX).
  • Premier scan = telechargement DB (~30 Mo), ensuite cache local.

5 - Scan IaC & Misconfigurations

0/5

Scan de misconfigurations : Trivy ne cherche pas seulement des vulnerabilites connues (CVE) mais aussi des erreurs de configuration. Un Dockerfile qui execute en root, un manifest Kubernetes sans limites de ressources, un bucket S3 public — ce sont des misconfigurations qui ouvrent des failles de securite.

Fichiers IaC supportes : Trivy scanne les fichiers Terraform (.tf), CloudFormation (YAML/JSON), Kubernetes manifests (YAML), Dockerfiles, et Helm charts. Commande : trivy config . ou trivy fs --scanners misconfig .

Frameworks de conformite : Trivy peut verifier la conformite avec des standards reconnus : CIS Benchmarks (Docker, Kubernetes), NSA/CISA Kubernetes Hardening Guide, AWS/Azure/GCP best practices. Chaque regle est mappee a un controle de conformite specifique.

Policies personnalisees (Rego/OPA) : En plus des regles integrees, on peut ecrire ses propres politiques de securite en Rego (le langage d'Open Policy Agent). Exemple : interdire les images Docker sans tag specifique, imposer des labels obligatoires sur les pods Kubernetes.

Scan Kubernetes en live : Avec trivy k8s --report summary cluster, Trivy peut scanner un cluster Kubernetes en direct et detecter les workloads vulnerables, les secrets exposes, les RBAC trop permissifs. C'est un audit de securite complet du cluster.

  • trivy config . = scanner les misconfigurations IaC.
  • Supporte : Terraform, CloudFormation, K8s YAML, Dockerfile, Helm.
  • CIS Benchmarks = standard de conformite de reference.
  • Rego/OPA = langage pour policies personnalisees.
  • trivy k8s = audit securite d'un cluster en live.
  • Misconfig ≠ CVE : erreur de config, pas vulnerabilite connue.

6 - Integration CI/CD & Bonnes Pratiques

0/5

Integration GitHub Actions : Trivy s'integre nativement dans GitHub Actions avec l'action officielle aquasecurity/trivy-action. On peut scanner les images buildees dans le workflow, echouer le pipeline si des vulnerabilites critiques sont detectees, et publier les resultats dans l'onglet Security de GitHub (format SARIF).

Integration Jenkins : Dans un Jenkinsfile, Trivy s'execute comme une etape shell : sh 'trivy image --exit-code 1 --severity CRITICAL myapp:${BUILD_TAG}'. Le flag --exit-code 1 fait echouer le build si des vulnerabilites sont trouvees. On peut aussi utiliser le plugin HTML Publisher pour archiver les rapports.

Le flag --exit-code : C'est le flag le plus important pour le CI/CD. --exit-code 0 (defaut) = Trivy rapporte mais ne bloque pas. --exit-code 1 = Trivy retourne un code d'erreur si des vulnerabilites sont trouvees, ce qui echoue le pipeline. Combine avec --severity pour ne bloquer que sur CRITICAL/HIGH.

Cache de la base de donnees : En CI/CD, telecharger la DB a chaque build ralentit le pipeline. Solutions : utiliser le cache CI (actions/cache dans GitHub Actions), monter un volume persistant, ou utiliser --skip-db-update avec une DB pre-telechargee. Trivy supporte aussi un serveur de DB centralise.

Formats de sortie : Trivy supporte plusieurs formats de rapport : table (terminal, lisible), JSON (parseable par des outils), SARIF (standard pour les outils de securite, compatible GitHub/Azure DevOps), template (personnalisable avec Go templates). Choix du format : --format json, --format sarif.

  • --exit-code 1 = echouer le pipeline si vulnerabilites trouvees.
  • GitHub Actions : aquasecurity/trivy-action (officiel).
  • SARIF = format standard pour resultats securite (GitHub Security).
  • Cache DB en CI pour eviter le telechargement a chaque build.
  • Formats : table, JSON, SARIF, template Go.
  • Combiner --exit-code 1 + --severity CRITICAL,HIGH = bloquer intelligemment.

Trivy dans un Jenkinsfile

pipeline { agent any stages { stage('Build') { steps { sh 'docker build -t myapp:${BUILD_TAG} .' } } stage('Security Scan') { steps { sh ''' trivy image --exit-code 1 \ --severity CRITICAL,HIGH \ --format table \ myapp:${BUILD_TAG} ''' } } stage('Deploy') { when { expression { currentBuild.result == null } } steps { sh 'docker push myapp:${BUILD_TAG}' } } } }

Trivy dans GitLab CI

security_scan: stage: test image: aquasec/trivy:latest script: - trivy image --exit-code 1 --severity CRITICAL,HIGH --format json --output trivy-report.json $CI_REGISTRY_IMAGE:$CI_COMMIT_SHA artifacts: paths: - trivy-report.json when: always

Bonnes pratiques en production

  • Scanner a chaque build — pas seulement periodiquement
  • Bloquer sur CRITICAL — avertir sur HIGH, ignorer LOW
  • Mettre a jour la DB quotidiennement — nouvelles CVE chaque jour
  • Utiliser des images minimales — Alpine, distroless, scratch
  • Scanner les images en registre — meme apres deploiement
  • Generer un SBOM — conformite et tracabilite
  • Automatiser les PR de mise a jour — Renovate, Dependabot
  • Combiner avec SonarQube — securite code + securite deps

Trivy Operator (Kubernetes)

  • Deploie Trivy comme un operateur K8s
  • Scan continu de tous les workloads du cluster
  • Rapports sous forme de CRDs (VulnerabilityReport, ConfigAuditReport)
  • Integration avec Prometheus pour les alertes
  • Installation : helm install trivy-operator aqua/trivy-operator
