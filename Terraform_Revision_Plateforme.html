1 - Pourquoi l'Infrastructure as Code ?

0/5

Le probleme de l'infrastructure manuelle : Configurer des serveurs manuellement (cliquer dans la console AWS, taper des commandes SSH) est lent, sujet aux erreurs, et non reproductible. Si un serveur tombe, reconstruire l'environnement identique prend des heures. Deux environnements "identiques" finissent toujours par diverger (configuration drift).

Configuration Drift : Le drift est la divergence progressive entre l'etat reel de l'infrastructure et l'etat desire. Un administrateur fait un correctif "temporaire" en production, un autre modifie une regle firewall — au bout de quelques mois, personne ne sait exactement ce qui tourne. L'IaC elimine ce probleme en definissant l'etat desire dans du code.

Infrastructure as Code (IaC) : L'IaC consiste a decrire toute l'infrastructure dans des fichiers de configuration versionnes. Les serveurs, reseaux, bases de donnees, permissions — tout est du code. On peut versionner (git), reviewer (PR), tester, et reproduire l'infrastructure a l'identique en quelques minutes.

Avantages de l'IaC : Reproductibilite (meme infra en dev/staging/prod), versionning (historique des changements), automatisation (CI/CD), documentation vivante (le code EST la documentation), collaboration (PR et code review), rapidite (deploiement en minutes au lieu d'heures).

Le role de Terraform : Terraform est l'outil IaC le plus populaire. Il permet de provisionner et gerer l'infrastructure chez n'importe quel provider cloud (AWS, Azure, GCP, Kubernetes, etc.) avec un seul langage (HCL). C'est l'approche "multi-cloud" par excellence.

  • Infrastructure manuelle = lente, fragile, non reproductible.
  • Configuration Drift = divergence entre etat reel et desire.
  • IaC = infrastructure decrite dans du code versionne.
  • Avantages : reproductibilite, versionning, automatisation, rapidite.
  • Terraform = outil IaC multi-cloud le plus populaire.
  • Le code EST la documentation de l'infrastructure.

2 - Presentation de Terraform

0/5

HashiCorp : Terraform est developpe par HashiCorp, une entreprise americaine fondee en 2012. HashiCorp cree des outils d'infrastructure : Terraform (IaC), Vault (secrets), Consul (service mesh), Nomad (orchestration), Packer (images), Vagrant (dev envs). Terraform est leur produit phare.

Open Source et licences : Terraform etait sous licence MPL 2.0 (open source) jusqu'en 2023, puis est passe sous BSL (Business Source License). En reaction, la communaute a cree OpenTofu (fork open source sous Linux Foundation). Terraform Cloud/Enterprise sont les versions commerciales.

Declaratif vs Imperatif : Terraform utilise l'approche declarative : on decrit l'ETAT FINAL desire ("je veux 3 serveurs"), pas les etapes pour y arriver ("cree serveur 1, cree serveur 2..."). Terraform calcule automatiquement les actions necessaires pour atteindre cet etat. Ansible est plutot imperatif (on decrit les etapes).

Terraform vs alternatives : CloudFormation = AWS uniquement. Ansible = imperatif, pas d'etat. Pulumi = IaC en langages generaux (Python, Go, TS). Terraform = declaratif, multi-cloud, ecosysteme le plus riche (4000+ providers). Chaque outil a ses forces.

Architecture : Terraform se compose du CLI (binaire unique), du langage HCL, des providers (plugins pour chaque service), et du state (fichier JSON qui stocke l'etat reel). Le workflow est simple : Write (code) → Plan (preview) → Apply (execution).

  • HashiCorp (USA, 2012) = createur de Terraform.
  • Declaratif = on decrit l'etat final, pas les etapes.
  • Multi-cloud : AWS, Azure, GCP, K8s, 4000+ providers.
  • OpenTofu = fork open source (Linux Foundation).
  • Workflow : Write → Plan → Apply.
  • Alternatives : CloudFormation (AWS), Ansible (imperatif), Pulumi (langages generaux).

3 - HCL & Syntaxe de base

0/6

HCL (HashiCorp Configuration Language) : HCL est le langage de configuration propre a Terraform. Il est concu pour etre lisible par les humains tout en etant parseable par les machines. L'extension des fichiers est .tf. HCL est un langage declaratif structure en blocs.

Structure d'un bloc : Un bloc HCL a la forme : type "label1" "label2" { arguments }. Exemple : resource "aws_instance" "web" { ami = "ami-123" }. Les types de blocs principaux sont : resource, data, variable, output, provider, terraform, locals, module.

Types de donnees : HCL supporte : string ("hello"), number (42, 3.14), bool (true/false), list (["a","b","c"]), map ({key="value"}), object, tuple, set. On peut aussi utiliser des expressions : interpolation ("${var.name}"), conditions (condition ? true : false), boucles (for).

terraform init : Commande obligatoire au debut. Elle initialise le repertoire de travail : telecharge les providers necessaires, configure le backend pour le state, installe les modules. A executer apres chaque ajout de provider ou module.

terraform plan : Genere un plan d'execution SANS rien modifier. Affiche ce qui sera cree (+), modifie (~), ou detruit (-). C'est une preview des changements. Indispensable pour verifier avant d'appliquer. Equivalent d'un "dry-run".

terraform apply / destroy : apply execute le plan et provisionne l'infrastructure reellement. Demande confirmation (ou -auto-approve en CI). destroy supprime TOUTES les ressources gerees par Terraform. Attention : irreversible !

  • HCL = langage declaratif de Terraform, fichiers .tf
  • Blocs : resource, data, variable, output, provider, module.
  • Types : string, number, bool, list, map, object.
  • terraform init = initialiser (providers, backend, modules).
  • terraform plan = preview sans modification (dry-run).
  • terraform apply = executer, destroy = tout supprimer.

4 - Providers

0/5

Qu'est-ce qu'un Provider ? Un provider est un plugin qui permet a Terraform d'interagir avec une API externe. Chaque service (AWS, Azure, GCP, Docker, Kubernetes, GitHub, Cloudflare...) a son propre provider. Le provider traduit les blocs HCL en appels API vers le service cible.

Terraform Registry : Le Registry (registry.terraform.io) est le depot officiel des providers et modules. On y trouve 4000+ providers (officiels par HashiCorp, verifies par les vendeurs, communautaires). Chaque provider a sa documentation avec les resources et data sources disponibles.

Configuration d'un provider : On declare un provider dans un bloc provider : credentials, region, version. Exemple AWS : provider "aws" { region = "eu-west-1" }. Les credentials sont generalement configurees via variables d'environnement (AWS_ACCESS_KEY_ID) et non en dur dans le code.

Versioning des providers : On controle la version du provider dans le bloc required_providers avec des contraintes : version = "~> 5.0" (compatible 5.x), >= 4.0, < 6.0". Cela evite les breaking changes inattendus lors d'un terraform init.

Providers multiples (alias) : On peut configurer plusieurs instances du meme provider avec des alias. Exemple : deployer dans 2 regions AWS differentes. provider "aws" { alias = "us" region = "us-east-1" } puis provider = aws.us dans la resource.

  • Provider = plugin pour interagir avec une API (AWS, Azure, K8s...).
  • Registry = depot officiel (4000+ providers).
  • Credentials via env vars, JAMAIS en dur dans le code.
  • Versioning : ~> 5.0 = compatible 5.x.
  • Alias = plusieurs configs du meme provider (multi-region).
  • terraform init telecharge les providers declares.

5 - Resources & Data Sources

0/5

Resource : Un bloc resource declare un element d'infrastructure a creer et gerer. Syntaxe : resource "type" "nom" { ... }. Exemple : resource "aws_instance" "web" { ami = "ami-123" instance_type = "t3.micro" }. Terraform cree, met a jour et supprime cette ressource.

Lifecycle : Chaque ressource a un cycle de vie : Create, Read, Update, Delete (CRUD). On peut personnaliser avec le bloc lifecycle : prevent_destroy = true (empecher la suppression), create_before_destroy = true (creer la nouvelle avant de supprimer l'ancienne), ignore_changes (ignorer certains attributs).

depends_on : Terraform detecte automatiquement les dependances via les references (ex: vpc_id = aws_vpc.main.id). Mais parfois les dependances sont implicites — on utilise alors depends_on = [aws_iam_role.example] pour forcer l'ordre de creation.

Data Source : Un bloc data lit des informations EXISTANTES sans les creer. Exemple : recuperer l'AMI la plus recente, lire un secret depuis Vault, obtenir l'ID d'un VPC existant. Syntaxe : data "aws_ami" "latest" { filter { ... } }. Lecture seule.

Resource Addressing : Chaque ressource a une adresse unique : type.nom (ex: aws_instance.web). On reference les attributs avec la notation point : aws_instance.web.id, aws_instance.web.public_ip. Ces references creent des dependances implicites.

  • resource = creer et gerer un element d'infrastructure.
  • Lifecycle : prevent_destroy, create_before_destroy, ignore_changes.
  • depends_on = forcer une dependance explicite.
  • data = lire des infos existantes (lecture seule).
  • Adressage : type.nom.attribut (ex: aws_instance.web.id).
  • References entre ressources = dependances implicites.

6 - Variables & Outputs

0/5

Input Variables : Permettent de parametrer la configuration. Declaration : variable "name" { type = string default = "value" description = "..." }. Utilisation : var.name. On peut les passer via CLI (-var), fichiers .tfvars, ou variables d'environnement (TF_VAR_name).

Types de variables : Primitifs : string, number, bool. Complexes : list(type), map(type), set(type), object({key=type}), tuple([types]). On peut ajouter des validations custom : validation { condition = ... error_message = "..." }.

Fichiers .tfvars : Les fichiers terraform.tfvars et *.auto.tfvars sont charges automatiquement. On peut aussi specifier un fichier : terraform apply -var-file="prod.tfvars". Pratique pour separer les valeurs par environnement (dev.tfvars, staging.tfvars, prod.tfvars).

Output Values : Les outputs exposent des valeurs apres apply : output "ip" { value = aws_instance.web.public_ip }. Utiles pour afficher des infos, passer des valeurs entre modules, ou servir d'input a d'autres outils. Commande : terraform output.

Locals : Les locals sont des variables internes calculees, non exposees en input. Utiles pour eviter la repetition : locals { common_tags = { env = var.env, project = "myapp" } }. Reference : local.common_tags.

  • variable = input parametrable (var.name).
  • Types : string, number, bool, list, map, object.
  • .tfvars = fichier de valeurs par environnement.
  • output = exposer des valeurs apres apply.
  • locals = variables internes calculees (local.xxx).
  • TF_VAR_name = passer une variable via env var.

7 - State Management

0/5

terraform.tfstate : Le state est un fichier JSON qui enregistre l'etat reel de l'infrastructure geree par Terraform. Il mappe chaque ressource du code a son equivalent reel (IDs, IPs, etc.). Sans le state, Terraform ne sait pas ce qu'il a deja cree et recrerait tout a chaque apply.

Remote Backends : Par defaut, le state est local (terraform.tfstate). En equipe, on utilise un remote backend : S3+DynamoDB (AWS), Azure Blob Storage, GCS, Consul, Terraform Cloud. Avantages : partage entre collegues, versioning, chiffrement, state locking.

State Locking : Le locking empeche deux personnes d'executer terraform apply en meme temps (ce qui corromprait le state). Avec S3, on utilise une table DynamoDB pour le lock. Terraform Cloud/Enterprise gere le locking automatiquement.

Commandes terraform state : terraform state list = lister les ressources. terraform state show <addr> = voir les details. terraform state mv = renommer/deplacer. terraform state rm = retirer du state (sans supprimer la ressource reelle).

terraform import : Permet d'importer une ressource existante (creee manuellement ou par un autre outil) dans le state Terraform. Syntaxe : terraform import aws_instance.web i-1234567890. Apres l'import, il faut ecrire le code HCL correspondant.

  • State = source de verite sur l'infra reelle (JSON).
  • Remote backend = partager le state en equipe (S3, GCS, TF Cloud).
  • Locking = empecher les apply concurrents (DynamoDB).
  • terraform state list/show/mv/rm = gerer le state.
  • terraform import = importer une ressource existante.
  • JAMAIS committer le state dans git (contient des secrets).

8 - Modules

0/5

Qu'est-ce qu'un module ? Un module est un conteneur reutilisable de ressources Terraform. C'est un repertoire contenant des fichiers .tf. Tout repertoire Terraform est un module (le "root module"). On peut appeler d'autres modules pour eviter la duplication et standardiser l'infrastructure.

Structure d'un module : Un module bien structure contient : main.tf (ressources), variables.tf (inputs), outputs.tf (outputs). On l'appelle avec : module "name" { source = "./modules/vpc" vpc_cidr = "10.0.0.0/16" }.

Sources de modules : Un module peut venir de : un chemin local (./modules/xxx), le Terraform Registry (hashicorp/consul/aws), GitHub (github.com/org/repo), S3, GCS, ou HTTP. Les modules du Registry sont versionnes et documentes.

Input/Output entre modules : On passe des valeurs a un module via ses variables (inputs). Le module expose des resultats via ses outputs. Le module parent accede aux outputs avec : module.vpc.vpc_id. C'est le mecanisme de communication inter-modules.

Bonnes pratiques modules : Un module doit faire UNE chose (principe de responsabilite unique). Versionner les modules (tags git). Documenter les inputs/outputs. Ne pas imbriquer plus de 2-3 niveaux. Utiliser les modules du Registry quand possible (bien testes, maintenus).

  • Module = repertoire de fichiers .tf reutilisable.
  • Structure : main.tf + variables.tf + outputs.tf.
  • Sources : local, Registry, GitHub, S3, HTTP.
  • Communication : variables (in) et outputs (out).
  • module.name.output = acceder aux outputs d'un module.
  • Un module = une responsabilite. Versionner avec des tags.

9 - Bonnes Pratiques & Production

0/5

Workspaces : Les workspaces permettent de gerer plusieurs environnements (dev, staging, prod) avec le meme code. Chaque workspace a son propre state. Commandes : terraform workspace new dev, terraform workspace select prod. Alternative : repertoires separes par env.

Integration CI/CD : En production, on n'execute JAMAIS terraform apply manuellement. Le workflow type : PR → terraform plan (affiche dans la PR) → review → merge → terraform apply automatique. Outils : GitHub Actions, GitLab CI, Jenkins, Atlantis (specialise Terraform).

Gestion des secrets : Ne JAMAIS stocker de secrets dans les fichiers .tf ou .tfvars commites. Solutions : variables d'env (TF_VAR_xxx), HashiCorp Vault, AWS Secrets Manager, les secrets du CI/CD. Le state aussi contient des secrets (d'ou remote backend chiffre).

Terraform Cloud / Enterprise : Terraform Cloud est la plateforme SaaS de HashiCorp : remote execution, state management, policy enforcement (Sentinel), collaboration equipe, private registry. Enterprise = version on-premise. Alternative open source : Atlantis + remote state.

Sentinel Policies : Sentinel est le framework de "policy as code" de HashiCorp. Il permet d'imposer des regles : "tous les buckets S3 doivent etre chiffres", "pas d'instances plus grandes que t3.large en dev", "tags obligatoires". Les policies bloquent le apply si violees.

  • Workspaces = multi-env avec le meme code (dev/staging/prod).
  • CI/CD : plan dans la PR, apply apres merge. Jamais de apply manuel en prod.
  • Secrets : env vars, Vault, Secrets Manager. JAMAIS en dur.
  • Terraform Cloud = SaaS (remote exec, Sentinel, collaboration).
  • Sentinel = policy as code (regles imposees avant apply).
  • Atlantis = alternative open source pour CI/CD Terraform.

CI/CD avec GitHub Actions

name: Terraform on: pull_request: paths: ['infra/**'] push: branches: [main] jobs: plan: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: hashicorp/setup-terraform@v3 - run: terraform init - run: terraform plan -no-color id: plan - uses: actions/github-script@v7 with: script: | github.rest.issues.createComment({ body: '```\n' + '${{ steps.plan.outputs.stdout }}' + '\n```' }) apply: if: github.ref == 'refs/heads/main' needs: plan runs-on: ubuntu-latest steps: - run: terraform apply -auto-approve
