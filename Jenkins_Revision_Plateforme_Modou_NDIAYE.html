1 - Problematique CI/CD

0/6

Le chaos sans automatisation : Imaginez une equipe de 5 developpeurs qui codent chacun de leur cote pendant des semaines. Au moment de fusionner leur travail, c'est le chaos : des dizaines de conflits, du code incompatible, des bugs impossibles a tracer. Les tests sont faits a la main (souvent oublies) et le deploiement est un moment de stress intense.

Integration Continue (CI) : La CI resout ce probleme. A chaque fois qu'un developpeur pousse son code (git push), un serveur automatique compile le projet, lance tous les tests et analyse la qualite du code. Si quelque chose casse, l'equipe est alertee en minutes, pas en semaines.

Deploiement Continu (CD) : Le CD va plus loin. "Continuous Delivery" signifie que le code est toujours pret a etre deploye en un clic (un humain valide). "Continuous Deployment" signifie que le deploiement en production est 100% automatique, sans aucune intervention humaine.

Pourquoi Jenkins ? Jenkins est l'outil CI/CD le plus utilise au monde. Il offre une interface web visuelle pour voir l'etat de tous vos projets, un historique complet de chaque build, des logs detailles, des notifications (email, Slack), plus de 1800 plugins et il est gratuit (open source) et auto-heberge (vos donnees restent chez vous).

  • CI = compiler + tester a chaque commit. Erreurs detectees immediatement.
  • CD = livrer/deployer automatiquement. Delivery = un clic. Deployment = zero humain.
  • Sans CI : conflits, tests oublies, deploiements stressants.
  • Jenkins = orchestrateur central CI/CD. Du commit au deploiement.
  • "Integrer souvent, integrer tot." Conflits petits et faciles.
  • Open source, 1800+ plugins, auto-heberge.

2 - Presentation Jenkins

0/6

Definition : Jenkins est un serveur open source ecrit en Java. Cree en 2004 sous le nom Hudson par Kohsuke Kawaguchi (Sun Microsystems), il a ete renomme Jenkins en 2011 suite a un conflit avec Oracle.

Architecture Master/Agent : Le Master est le serveur central qui gere l'interface web, planifie les taches et orchestre tout. Les Agents sont des machines distantes qui executent les builds. Ils communiquent via SSH (le master initie) ou JNLP (l'agent initie).

Job : Une tache configuree dans Jenkins (ex: compiler un projet, lancer des tests). C'est l'unite de travail de base.

Build : Une execution d'un job. Chaque fois qu'un job se lance, ca cree un build numerote (#1, #2, etc.) avec ses logs et son resultat (succes/echec).

Pipeline : Un enchainement d'etapes automatisees ecrit en code Groovy (dans un fichier Jenkinsfile). C'est la facon moderne de definir son CI/CD : tout est versionne dans Git.

Workspace : Le dossier temporaire sur l'agent ou Jenkins clone le code source et execute les commandes. Chaque job a son propre workspace.

Artifacts : Les fichiers produits par un build (ex: un .jar Java, un .zip, une image Docker). Ils peuvent etre archives et telecharges depuis Jenkins.

Ecosysteme : Jenkins s'integre avec Git (code), Docker (conteneurs), Kubernetes (orchestration), AWS/Azure (cloud), Ansible/Terraform (infra), SonarQube (qualite code).

  • Java, open source, +300K installations.
  • Master/Agent = scalabilite horizontale.
  • Pipeline = Jenkinsfile en Groovy, versionne.
  • 1800+ plugins, 600+ contributeurs.
  • LTS en production, Weekly en dev.
  • Du commit au deploiement.

3 - Installation

0/6

Prerequis : Jenkins a besoin de Java (version 11 ou 17) installe sur la machine. Il utilise le port 8080 pour son interface web. Au premier lancement, un mot de passe unique (initialAdminPassword) est genere pour securiser l'acces.

Installation Docker (recommandee) : Docker permet de lancer Jenkins dans un conteneur isole, sans polluer votre systeme. La commande : docker run -p 8080:8080 -p 50000:50000 -v jenkins_home:/var/jenkins_home jenkins/jenkins:lts-jdk17. Le "-v" cree un volume pour persister vos donnees meme si le conteneur est supprime.

Installation WAR : Alternative sans Docker. On telecharge le fichier jenkins.war et on le lance avec : java -jar jenkins.war --httpPort=8080. Plus simple mais moins isole.

Post-installation : Apres le lancement, ouvrez localhost:8080 dans le navigateur. Collez le mot de passe initial (affiche dans les logs ou dans le fichier secrets/initialAdminPassword). Choisissez "Install suggested plugins" pour avoir les essentiels. Enfin, creez votre compte administrateur.

  • Java 11/17 requis.
  • Port 8080 (web), 50000 (JNLP agents).
  • Docker recommande.
  • Volume jenkins_home persiste tout.
  • initialAdminPassword pour premier acces.
  • Suggested plugins = essentiels.

4 - Configuration

0/5

JENKINS_HOME : C'est LE dossier vital de Jenkins (par defaut /var/lib/jenkins). Il contient tout : les jobs (vos projets), les plugins installes, les workspaces (code en cours), les secrets (mots de passe chiffres) et les logs. Perdre ce dossier = perdre toute votre configuration.

Securite : Jenkins offre plusieurs modes de securite. "Matrix-based security" permet de definir precisement qui peut faire quoi (lire, construire, administrer). ATTENTION : le mode "Anyone can do anything" desactive TOUTE securite — ne JAMAIS l'utiliser en production.

Credentials : Systeme securise pour stocker mots de passe, cles SSH, tokens API. Ils sont chiffres (AES-128), injectes dans les builds via withCredentials() et automatiquement masques dans les logs (affiches comme ****).

Variables d'environnement : Des valeurs accessibles dans tout le pipeline via env.NOM (ex: env.BUILD_NUMBER, env.GIT_BRANCH). On peut definir les siennes dans le bloc environment {} du Jenkinsfile.

  • JENKINS_HOME = tout. Backup critique.
  • JAMAIS "Anyone can do anything".
  • Matrix-based = controle fin.
  • Credentials chiffres.
  • URL publique pour webhooks.

0/5

Jenkins sans plugins : Jenkins "nu" est tres basique. Ce sont les plugins qui lui donnent ses super-pouvoirs : connexion a Git, integration Docker, notifications Slack, etc. Plus de 1800 plugins sont disponibles gratuitement.

6 - Pipelines

0/6

Pipeline as Code : Au lieu de configurer votre CI/CD a la souris dans Jenkins (fragile, non versionne), on ecrit un fichier "Jenkinsfile" en langage Groovy qu'on place a la racine de son depot Git. Ainsi, votre pipeline est versionne, revocable et partageable comme du code.

Declaratif vs Scripte : Il existe 2 syntaxes. Le Declaratif (95% des cas) est structure et simple : pipeline { agent... stages... post... }. Le Scripte est du Groovy pur avec node {} — plus flexible mais plus complexe, reserve aux cas avances.

Structure d'un pipeline : pipeline { agent (OU executer), stages { stage('Nom') { steps { commandes } } } (QUOI faire), post { actions apres } (notifications, cleanup) }. Chaque stage est une etape logique (Build, Test, Deploy).

Parallelisme : Le bloc parallel {} execute plusieurs stages en meme temps (ex: tester sur Linux ET Windows simultanement). @Library permet de reutiliser du code pipeline commun entre projets (DRY = Don't Repeat Yourself).

  • Jenkinsfile a la racine Git.
  • Declaratif 95% des cas.
  • Stage = etape logique nommee.
  • parallel {} = simultane.
  • post {} = notifications.
  • Toujours un timeout.

7 - Integration Git

0/5

3 facons de declencher un build : 1) Webhook (ideal) : GitHub previent Jenkins instantanement a chaque push. 2) Polling SCM : Jenkins interroge Git periodiquement (toutes les X minutes) — consomme des ressources. 3) Manuel : cliquer "Build Now" a la main.

Webhook — comment ca marche : Dans les parametres GitHub de votre repo, vous ajoutez l'URL JENKINS_URL/github-webhook/. A chaque push, GitHub envoie une notification HTTP a Jenkins qui demarre le build immediatement. Cote Jenkins, activez "GitHub hook trigger for GITScm polling".

Authentification SSH : Pour que Jenkins accede a un repo prive, on genere une paire de cles SSH (ssh-keygen -t ed25519). La cle publique va sur GitHub, la cle privee dans Jenkins (Credentials). ed25519 est l'algorithme le plus moderne et securise.

Multibranch Pipeline : Un type special de job qui detecte automatiquement TOUTES les branches contenant un Jenkinsfile. Chaque branche obtient son propre build. Branches supprimees = jobs supprimes automatiquement. Ideal pour les equipes avec beaucoup de feature branches.

  • Webhook = ideal, zero delai.
  • Polling = secours (pas d'acces Internet).
  • Cles privees JAMAIS en clair.
  • Multibranch = auto detection.
  • ed25519 recommande.

8 - Jenkins + Docker

0/5

Pourquoi Docker + Jenkins : Docker garantit que votre build s'execute dans un environnement IDENTIQUE a chaque fois (meme OS, memes versions). Plus de "ca marche sur ma machine". Le plugin Docker Pipeline permet d'utiliser des images Docker comme agents de build.

Agent Docker : Avec agent { docker { image 'python:3.11-slim' } }, Jenkins lance un conteneur avec Python 3.11, execute vos commandes dedans, puis le detruit. Chaque stage peut avoir une image differente. docker.sock monte permet au conteneur d'executer des commandes Docker (build d'images).

Construire des images : On utilise withCredentials() pour s'authentifier au registry (Docker Hub, ECR...) sans exposer le mot de passe. Le tag BUILD_NUMBER (ex: monapp:42) identifie chaque version. Le "multi-stage build" Dockerfile produit des images legeres en separant compilation et execution.

Trivy — Scanner de securite : Trivy analyse vos images Docker pour detecter les vulnerabilites connues (CVE). Integre dans le pipeline, il bloque le deploiement si des failles critiques sont trouvees. Gratuit et open source.

  • Docker = reproductibilite.
  • Chaque stage = container different.
  • docker.sock = DinD.
  • Tag BUILD_NUMBER.
  • Multi-stage = leger.

9 - Notifications

0/5

Notifications par email : Le plugin "Email Extension" (emailext) envoie des emails riches en HTML quand un build echoue ou reussit. On l'utilise dans le bloc post {} du pipeline : post { failure { emailext(...) } }. On peut personnaliser les destinataires, le sujet et le corps du message.

Notifications Slack : slackSend() envoie un message dans un canal Slack en temps reel. Les couleurs indiquent le statut : "good" (vert = succes), "warning" (orange = instable), "danger" (rouge = echec). L'equipe est alertee instantanement sans verifier Jenkins.

Autres canaux : Microsoft Teams (plugin Office 365 Connector), Webhooks generiques (n'importe quel service), PagerDuty (alerte l'equipe on-call en cas de panne critique). On choisit selon les outils de l'equipe.

Dashboard et historique : Blue Ocean offre une visualisation graphique des pipelines en temps reel. L'historique des builds permet de voir les tendances (builds qui cassent souvent = code fragile). buildDiscarder limite le nombre de builds conserves pour ne pas saturer le disque.

  • Notifications = reagir vite.
  • emailext() = HTML riche.
  • slackSend() = temps reel.
  • good/warning/danger.
  • buildDiscarder pour espace.

10 - Deploiement

0/5

L'enjeu du deploiement : Quand vous mettez a jour votre application en production, les utilisateurs ne doivent RIEN remarquer. Pas de page d'erreur, pas de coupure. C'est le "zero downtime" (zero interruption). 4 strategies principales existent pour y arriver.

Blue/Green : Vous avez 2 environnements identiques : "Blue" (actuel, en ligne) et "Green" (nouveau, en preparation). Quand Green est pret et teste, vous basculez le trafic instantanement. Si ca casse, vous rebasculez vers Blue en secondes (rollback instantane). Inconvenient : coute 2x en infrastructure.

Canary (le canari dans la mine) : On envoie d'abord ~5% du trafic vers la nouvelle version. On observe les erreurs, les temps de reponse. Si tout va bien, on augmente progressivement (10%, 25%, 50%, 100%). Si probleme, on redirige les 5% vers l'ancienne version. Tres sur mais lent.

Rolling Update : Les instances (serveurs/pods) sont mises a jour une par une. A tout moment, certaines tournent sur l'ancienne version et d'autres sur la nouvelle. C'est la strategie native de Kubernetes. Simple mais pas de rollback instantane.

  • Blue/Green = rollback instant, 2x cout.
  • Canary = progressif, 5%.
  • Rolling = natif K8s.
  • Health check obligatoire.
  • Recreate = JAMAIS en prod.

11 - Jenkins vs GH Actions

0/5

Jenkins : Vous l'installez sur VOS serveurs (auto-heberge). Avantages : controle total des donnees, personnalisation infinie via 1800+ plugins, pas de limite de minutes/mois. Inconvenients : vous gerez la maintenance, les mises a jour, la securite et l'infrastructure. Pipelines en Groovy.

GitHub Actions : Service cloud integre a GitHub (SaaS = Software as a Service). Avantages : zero maintenance, zero installation, demarre en 5 minutes, workflows en YAML simple. Inconvenients : depend de GitHub, minutes limitees (plan gratuit), moins personnalisable, donnees chez Microsoft.

Approche hybride : Le meilleur des deux mondes. GitHub Actions pour le CI leger (linter, tests rapides sur les Pull Requests) et Jenkins pour le CD complexe (deploiement multi-environnement, pipelines longs, besoin de machines specifiques). Actions declenche Jenkins via son API REST.

Autres alternatives : GitLab CI (integre a GitLab, YAML), CircleCI (SaaS rapide), Argo CD (deploiement GitOps natif Kubernetes), Tekton (pipelines comme ressources Kubernetes). Chaque outil a ses forces — il n'y a pas de solution universellement meilleure.

  • Jenkins = controle total + maintenance.
  • Actions = zero maintenance - controle.
  • Pas d'outil universellement meilleur.
  • Hybride = best of both.
  • Argo/Tekton = natif K8s.

12 - Projet fil rouge

0/5

Pipeline complet — les 5 etapes : 1) Checkout : Jenkins clone le code depuis Git. 2) Install : installation des dependances (pip install -r requirements.txt). 3) Test : execution des tests automatises (pytest) avec rapport JUnit XML. 4) Build Docker : creation d'une image Docker taguee avec le numero de build. 5) Deploy : deploiement en production (seulement depuis la branche main).

Agent Docker : On utilise l'image python:3.11-slim comme environnement d'execution. "-slim" = version allegee (200 Mo au lieu de 900 Mo), suffisante pour la plupart des projets Python. Les tests generent un rapport JUnit XML que Jenkins sait parser et afficher.

Deploiement conditionnel : La directive when { branch 'main' } signifie : "n'execute ce stage QUE si on est sur la branche main". Les feature branches (feature/login, fix/bug-42) passent par Build et Test mais ne deploient PAS. Cela protege la production des codes non valides.

Strategie Blue/Green : Deux environnements identiques tournent. Le nouveau code est deploye sur l'environnement inactif. Un health check verifie que tout fonctionne. Si OK, le load balancer bascule le trafic. Si probleme apres bascule, rollback en re-pointant vers l'ancien environnement (< 1 seconde).

  • Integre TOUTES les notions.
  • when main = deploy conditionnel.
  • Tag BUILD_NUMBER.
  • Health check avant bascule.
  • Notifications temps reel.

13 - Freestyle Jobs

0/5

Qu'est-ce qu'un Freestyle Job : C'est le type de job le plus ancien de Jenkins. On le configure entierement via l'interface web en cliquant et en remplissant des champs. Aucun code a ecrire. Parfait pour decouvrir Jenkins ou pour des taches simples (un script de backup, un rapport quotidien).

Les 5 sections de configuration : 1) General : nom, description, parametres. 2) Source Code Management (SCM) : URL Git, branche, credentials. 3) Build Triggers : quand declencher (webhook, cron, apres un autre job). 4) Build Steps : les commandes a executer. 5) Post-build Actions : archiver, notifier, declencher un autre job.

Build Steps disponibles : "Execute shell" (commandes Linux/Mac), "Windows batch command" (commandes Windows), "Invoke Maven" (projets Java Maven), "Invoke Gradle" (projets Java Gradle). On peut enchainer plusieurs steps.

Limites importantes : La config est stockee dans Jenkins (config.xml), PAS dans Git = non versionnee, risque de perte. Pas de parallel (stages simultanes). Pas de "when" (conditions). Pas de stages visuels. Pour tout projet serieux, preferez un Pipeline (Jenkinsfile).

  • UI web, zero code.
  • Non versionne = risque perte.
  • Pas de logique conditionnelle.
  • Bon pour prototypage/admin.
  • Migrer vers Pipeline si complexe.

14 - Jenkinsfile detail

0/5

Les directives (blocs obligatoires) : pipeline {} = le conteneur racine de tout. agent = ou ca s'execute. stages {} = liste des etapes. stage('Nom') = une etape nommee. steps {} = les commandes a l'interieur d'un stage. post {} = ce qui se passe apres (succes ou echec). environment {} = variables d'environnement.

Types d'agent : "any" = n'importe quel agent dispo (Jenkins choisit). "none" = pas d'agent global (chaque stage definit le sien). label('linux') = un agent avec ce label. docker { image '...' } = un conteneur Docker. kubernetes {} = un pod Kubernetes cree a la demande.

Directive when (conditions) : when { branch 'main' } = execute seulement sur main. when { expression { return env.DEPLOY == 'true' } } = condition Groovy libre. when { changeset '**/*.js' } = seulement si des fichiers .js ont change. Permet de sauter des stages inutiles.

Parametres (inputs utilisateur) : parameters { string(name:'VERSION'), booleanParam(name:'DEPLOY_PROD'), choice(name:'ENV', choices:['dev','staging','prod']) }. L'utilisateur remplit un formulaire avant le build. Acces dans le code via params.VERSION, params.DEPLOY_PROD, etc.

  • pipeline {} = racine obligatoire.
  • agent any = n'importe quel agent.
  • when = conditionnel.
  • params.NOM pour parametres.
  • input = approbation manuelle.

15 - Securite

0/5

Authentification (QUI es-tu ?) : 3 methodes pour verifier l'identite : 1) Jenkins interne = base de donnees locale (petites equipes). 2) LDAP/Active Directory = annuaire d'entreprise centralise (grands groupes, SSO). 3) SAML/OAuth = connexion via Google, GitHub, Okta (applications modernes).

Autorisation (QUE peux-tu faire ?) : Une fois identifie, que peut faire l'utilisateur ? "Matrix-based" = tableau permissions par utilisateur. "Role Strategy" (plugin) = roles reutilisables (dev, qa, admin) assignables a des groupes. "Project-based" = permissions par projet/dossier.

Gestion des secrets (Credentials) : 3 types : Username/Password (login classique), SSH Key (connexion serveurs/Git), Secret Text (tokens API, cles). 3 scopes de visibilite : Global (tous les jobs), System (agents et config systeme uniquement), Folder (un dossier de projets specifique).

Script Security et CSRF : "Script Security" execute les scripts Groovy dans une sandbox (bac a sable) — les methodes dangereuses (acces fichiers, reseau) doivent etre approuvees par un admin. CSRF (Cross-Site Request Forgery) empeche les sites malveillants d'executer des actions Jenkins a votre insu — TOUJOURS laisser active en production.

  • Auth != Autorisation.
  • LDAP/AD pour entreprises.
  • Role Strategy = roles reutilisables.
  • Script Security = sandbox.
  • CSRF TOUJOURS active.

16 - Agents & Noeuds

0/5

Vocabulaire essentiel : Node = une machine (physique ou VM) enregistree dans Jenkins. Agent = le processus Java qui tourne sur cette machine et communique avec le master. Executor = un "slot" d'execution sur un node. Si un node a 4 executors, il peut faire 4 builds en parallele.

2 types d'agents : 1) Permanents : machines dediees, toujours connectees. Connexion par SSH (le master se connecte a l'agent) ou JNLP/WebSocket (l'agent se connecte au master — utile si l'agent est derriere un firewall). 2) Cloud/Dynamiques : conteneurs Docker ou pods Kubernetes crees a la demande et detruits apres le build. Scalabilite automatique.

Labels (etiquettes) : Des tags qu'on attribue aux nodes pour les categoriser (ex: "linux", "windows", "docker", "gpu"). Dans le pipeline, agent { label 'linux && docker' } signifie "execute ce build sur un node qui a A LA FOIS les labels linux et docker". Permet de router les builds intelligemment.

Regle d'or du Master : Le master Jenkins doit avoir numExecutors = 0. Il ne doit JAMAIS executer de builds lui-meme. Pourquoi ? 1) Securite : un build malveillant pourrait compromettre tout Jenkins. 2) Performance : les builds consommeraient les ressources du master et ralentiraient l'interface pour toute l'equipe.

  • Node=machine, Agent=processus, Executor=slot.
  • Master = 0 executors.
  • Labels = router builds.
  • Cloud = scalabilite a la demande.
  • SSH=master initie, JNLP=agent initie.

17 - Troubleshooting

0/5

Ou trouver les logs : 3 niveaux de logs. 1) Console Output : les logs d'UN build specifique (premier reflexe quand un build echoue). 2) System Log (Manage Jenkins > System Log) : logs globaux de Jenkins. 3) Fichier /var/log/jenkins/jenkins.log : logs systeme, utiles si Jenkins plante completement.

Problemes frequents et solutions : OOM (Out of Memory) = Jenkins manque de RAM. Solution : augmenter -Xmx dans les options JVM (ex: -Xmx4g pour 4 Go). Disque plein = trop de builds conserves. Solution : buildDiscarder + workspace cleanup. Agent offline = machine ou processus down. Plugin incompatible = toujours tester sur staging avant mise a jour.

Script Console : Accessible via Manage Jenkins > Script Console. Permet d'executer du code Groovy DIRECTEMENT sur le master Jenkins (lister jobs, supprimer builds, modifier config). Extremement puissant — reserve aux administrateurs. Un script mal ecrit peut detruire votre instance.

Replay : Fonctionnalite geniale pour le debug. Depuis un build, cliquez "Replay" : vous pouvez modifier le Jenkinsfile temporairement et relancer le build SANS faire de commit Git. Ideal pour tester une correction rapidement. Le changement n'est pas sauvegarde — pensez a commiter une fois que ca marche.

  • Console Output = premier reflexe.
  • OOM = augmenter -Xmx.
  • Disque = logRotator + cleanup.
  • Replay = modifier sans commit.
  • Script Console = admin only.
