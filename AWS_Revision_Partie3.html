Amazon VPC

Qu'est-ce qu'un VPC ?

Amazon VPC (Virtual Private Cloud) est un reseau virtuel logiquement isole dans le cloud AWS. Vous definissez et controlez entierement la plage d'adresses IP, les sous-reseaux, les tables de routage et les passerelles.

Composants fondamentaux

ComposantRole
Sous-reseauSegment du VPC dans une AZ. Public ou prive.
Table de routageRegles determinant ou diriger le trafic.
Internet GatewayCommunication VPC <-> Internet.
NAT GatewayAcces Internet sortant pour sous-reseaux prives.
Security GroupPare-feu instance (stateful).
NACLPare-feu sous-reseau (stateless).
Elastic IPAdresse IPv4 statique publique.

Sous-reseaux : Public vs Prive

Public : Route vers l'IGW (0.0.0.0/0 -> IGW). Instances avec IP publique.

Prive : Pas de route directe vers Internet. Utilise un NAT Gateway pour le trafic sortant.

CIDR et Adressage

CIDR definit la plage d'adresses. VPC : entre /16 (65 536 IP) et /28 (16 IP).

AWS reserve 5 adresses par sous-reseau (4 premieres + derniere).

Exemple : 10.0.0.0/16 = adresses 10.0.0.0 a 10.0.255.255

Points Cles Enrichis
⭐
Un VPC = une Region, plusieurs AZ
Un VPC est limite a une seule Region mais peut contenir des sous-reseaux dans chaque AZ de cette Region.
⭐
Un sous-reseau = une seule AZ
Un sous-reseau ne peut jamais s'etendre sur plusieurs AZ. Pour la HA, creez des sous-reseaux dans plusieurs AZ.
⭐
Security Group = Stateful
Si le trafic entrant est autorise, le retour sortant est automatique. Pas besoin de regle sortante explicite pour les reponses.
⭐
NACL = Stateless
Regles entrant ET sortant doivent etre definies. Ports ephemeres (1024-65535) necessaires pour les reponses.
⭐
VPC par defaut dans chaque Region
Chaque compte AWS a un VPC par defaut avec un sous-reseau public dans chaque AZ et un IGW attache.
⭐
5 adresses reservees par sous-reseau
.0 (reseau), .1 (routeur), .2 (DNS), .3 (reserve), derniere (broadcast). Un /24 = 251 IP disponibles.
⭐
IGW = hautement disponible par defaut
L'Internet Gateway est redondant et scalable, pas besoin de le configurer en HA.

VPC Peering - Regles importantes

  • Non transitif : A-B et B-C ne donne PAS A-C
  • Les CIDR ne doivent pas se chevaucher
  • Fonctionne entre comptes et entre Regions
  • Le trafic reste sur le reseau prive AWS

VPC Endpoints : Gateway vs Interface

Gateway Endpoint : S3 et DynamoDB uniquement. Gratuit. Configure dans la table de routage.

Interface Endpoint (PrivateLink) : Autres services. Cree une ENI. Payant (horaire + donnees).

Direct Connect vs VPN

Direct Connect : Connexion physique dediee, bande passante constante, latence previsible. Mise en place en semaines.

VPN : Passe par Internet (chiffre IPSec), mise en place rapide, bande passante variable.

Points Cles Enrichis
⭐
VPC Peering = NON transitif
Chaque paire de VPC necessite sa propre connexion. Pour N VPC entierement connectes, il faut N*(N-1)/2 peerings.
⭐
Gateway Endpoint = gratuit (S3 + DynamoDB)
Se configure dans la table de routage. Interface Endpoint = payant, cree une ENI dans le sous-reseau.
⭐
Transit Gateway = hub central (hub-and-spoke)
Simplifie les architectures complexes. Evite le mesh complet de peering. Un seul point de gestion.
⭐
Direct Connect = semaines d'installation
Connexion physique (fibre). Bande passante 1 Gbps ou 10 Gbps. Pas chiffre par defaut (ajouter VPN par-dessus si besoin).
⭐
VPN = Virtual Private Gateway + Customer Gateway
Virtual Private Gateway cote AWS. Customer Gateway = votre equipement/logiciel on-premises. Deux tunnels pour la HA.

VPC Flow Logs

Capturent le trafic IP (metadonnees) des interfaces reseau.

  • Activables au niveau VPC, sous-reseau ou ENI
  • Publiables vers CloudWatch Logs ou S3
  • Ne capturent PAS le contenu des paquets
  • Montrent ACCEPT/REJECT pour diagnostiquer
Points Cles Enrichis
⭐
SG peut referencer un autre SG
Au lieu de specifier une IP, on peut autoriser le trafic provenant d'instances ayant un certain Security Group.
⭐
NACL : premiere correspondance gagne
Les regles sont evaluees du numero le plus bas au plus haut. Regle 100 avant 200. Regle * (deny all) en dernier.
⭐
Flow Logs = diagnostic de connectivite
Permettent de voir quel trafic est accepte ou rejete. Utile pour trouver pourquoi une connexion echoue.

Depannage d'un VPC

Problemes courants et solutions

ProblemeCause probableSolution
Instance ne peut pas acceder InternetPas de route IGW, pas d'IP publiqueVerifier route + IP publique/EIP
SSH impossiblePort 22 bloque (SG ou NACL)Ouvrir port 22 inbound
Instance privee sans InternetNAT Gateway absent ou mal placeNAT GW dans subnet PUBLIC
Peering ne fonctionne pasRoutes manquantesRoutes dans les DEUX VPC

Methodologie systematique

  • 1. Verifier Security Groups (inbound/outbound)
  • 2. Verifier NACLs (regles + ordre)
  • 3. Verifier tables de routage (IGW, NAT, Peering)
  • 4. Verifier adressage IP (IP publique, EIP)
  • 5. Consulter VPC Flow Logs (ACCEPT/REJECT)
  • 6. Verifier DNS (enableDnsHostnames/Support)
Points Cles Enrichis
⭐
NAT Gateway = sous-reseau PUBLIC obligatoire
Erreur la plus frequente : placer le NAT GW dans un sous-reseau prive. Il DOIT etre dans le public subnet avec route vers IGW.
⭐
Peering : routes dans LES DEUX VPC
Il faut ajouter une route vers le CIDR de l'autre VPC dans la table de routage de CHAQUE VPC.
⭐
SG ET NACL doivent autoriser le trafic
Le trafic doit passer les DEUX couches. Si l'un bloque, la connexion echoue meme si l'autre autorise.
⭐
Flow Logs : ACCEPT/REJECT pour diagnostiquer
Si vous voyez REJECT dans les Flow Logs, le trafic est bloque par un SG ou une NACL. Identifiez la source du blocage.

Amazon EBS

Qu'est-ce qu'Amazon EBS ?

Service de stockage bloc persistant pour EC2. Les donnees persistent independamment de la vie de l'instance. Comme un disque dur virtuel attache a votre instance.

Types de volumes

TypeIOPS maxCas d'usage
gp3/gp2 (SSD)16 000Usage general, boot
io2/io1 (SSD)256 000BDD critiques, I/O intensif
st1 (HDD)500Big data, throughput
sc1 (HDD)250Donnees froides

Snapshots

Sauvegardes incrementales stockees dans S3. Premier = copie complete, suivants = blocs modifies. Copiables entre Regions pour la DR.

Chiffrement

AWS KMS (AES-256). Chiffre donnees au repos + en transit + snapshots. Un volume non chiffre necessite : snapshot -> copie chiffree -> nouveau volume.

Points Cles Enrichis
⭐
EBS = lie a une AZ
Pour deplacer un volume : snapshot (au niveau Region) puis recreer dans l'autre AZ.
⭐
gp3 > gp2 (meilleur rapport qualite/prix)
gp3 permet de configurer IOPS et debit independamment. 20% moins cher que gp2 pour les memes perfs.
⭐
st1 et sc1 = PAS des volumes de boot
Seuls les SSD (gp2/gp3, io1/io2) peuvent servir de volume de demarrage OS.
⭐
Multi-Attach = io1/io2 uniquement
Permet a un volume d'etre attache a plusieurs instances dans la MEME AZ. Utile pour clusters.
⭐
DeleteOnTermination = actif par defaut pour root
Le volume root est supprime quand l'instance est terminee. Les volumes supplementaires sont conserves par defaut.
⭐
Snapshots incrementaux mais autonomes
Chaque snapshot est suffisant pour restaurer, meme si seuls les blocs modifies sont stockes.

Instance Store

Definition

Stockage bloc temporaire physiquement attache au serveur hote. Donnees PERDUES a l'arret, terminaison ou defaillance materielle. Performance I/O tres elevee.

Comparaison avec EBS

AspectInstance StoreEBS
PersistanceEphemerePersistant
PerformanceTres hauteHaute
CoutInclusSepare
SnapshotNonOui
Points Cles Enrichis
⭐
Ephemere = perdu au stop/terminate
Les donnees survivent au REBOOT mais pas au STOP, TERMINATE ou defaillance hardware.
⭐
Ideal pour caches et donnees temporaires
Buffers, fichiers scratch, donnees de calcul temporaires. Jamais pour des donnees critiques.
⭐
Pas de snapshot possible
C'est votre responsabilite de repliquer les donnees si necessaire.
⭐
Performance superieure a EBS
Disque physiquement sur le serveur, pas de latence reseau. Meilleur pour I/O intensif ephemere.

Amazon EFS

Qu'est-ce qu'EFS ?

Systeme de fichiers NFS elastique, monte simultanement par plusieurs instances EC2 Linux. S'adapte automatiquement en taille. Protocole NFSv4.1.

Classes de stockage

ClasseDescription
StandardMulti-AZ, acces frequent
Standard-IAMulti-AZ, acces rare
One Zone1 AZ, acces frequent
One Zone-IA1 AZ, acces rare
Points Cles Enrichis
⭐
EFS = multi-instances, multi-AZ, Linux
Difference clef avec EBS (1 instance, 1 AZ). Pour Windows : FSx for Windows (SMB).
⭐
Elastique = pas de provisionnement
La taille croit/decroit automatiquement. Vous payez uniquement le stockage utilise.
⭐
Lifecycle management vers IA
Deplace automatiquement les fichiers non accedes depuis X jours vers la classe IA (moins chere).
⭐
Plus cher que EBS mais partage
Le cout par Go est superieur a EBS, mais le partage entre instances justifie l'investissement.

Amazon S3

Stockage objet

Durabilite 99,999999999% (11 neufs). Objets dans des buckets. Acces HTTP/HTTPS. Taille max objet : 5 To. Noms de buckets globalement uniques.

Classes de stockage

ClasseDisponibiliteUsage
Standard99,99%Acces frequent
Standard-IA99,9%Peu frequent, acces rapide
One Zone-IA99,5%Reproductible, peu accede
Intelligent-Tiering99,9%Patterns imprevisibles
Glacier Instant99,9%Archive, acces ms
Glacier Flexible99,99%Archive, minutes/heures
Glacier Deep Archive99,99%Archive, 12-48h

Fonctionnalites cles

Versioning : Conserve toutes les versions. Lifecycle : Transition auto entre classes. Replication : CRR/SRR. Chiffrement : SSE-S3, SSE-KMS, SSE-C. Presigned URL : Acces temporaire.

Points Cles Enrichis
⭐
11 neufs de durabilite
99,999999999%. Perdre 1 objet sur 10 millions en 10 000 ans. Quasi impossible de perdre des donnees.
⭐
S3 n'est PAS un systeme de fichiers
Stockage objet acces via API HTTP. Ne peut PAS etre monte sur une instance comme un disque.
⭐
Noms de buckets = globalement uniques
Pas juste unique dans votre compte mais dans TOUT AWS. Choisissez des noms descriptifs et specifiques.
⭐
Upload > 5 Go = Multipart Upload obligatoire
Single PUT limite a 5 Go. Multipart Upload pour les objets plus gros (max 5 To).
⭐
Versioning : actif, ne peut etre que suspendu
Une fois active, on ne peut pas desactiver le versioning, seulement le suspendre.
⭐
Tous les buckets sont PRIVES par defaut
Block Public Access active par defaut. Il faut explicitement rendre un bucket public.

Amazon Glacier

Archivage a tres bas cout

Pour donnees rarement accedees, conservees longtemps. Cout stockage tres bas, recuperation a delai et cout.

Delais de recuperation

ClasseExpeditedStandardBulk
Instant RetrievalMillisecondes
Flexible Retrieval1-5 min3-5h5-12h
Deep ArchiveN/A12h48h

Vault Lock (WORM)

Politique immuable Write Once Read Many. Impossible de modifier/supprimer. Duree min : 90 jours (Flexible) / 180 jours (Deep Archive).

Points Cles Enrichis
⭐
Deep Archive = le moins cher d'AWS
~0.00099$/Go/mois. Mais recuperation en 12-48h. Ideal pour conformite 7-10+ ans.
⭐
Vault Lock = conformite immuable
WORM : une fois verrouille, personne (meme root) ne peut supprimer/modifier avant l'expiration.
⭐
Frais de suppression anticipee
Si vous supprimez avant 90j (Flexible) ou 180j (Deep Archive), des frais s'appliquent.

Amazon CloudWatch

Service de surveillance et observabilite

Collecte metriques, logs et evenements. Permet de surveiller, alerter et reagir automatiquement.

Composants

ComposantRole
MetriquesDonnees numeriques (CPU, reseau)
AlarmesDeclenchent actions si seuil depasse
LogsCollecte centralisee des logs
EventsReagit aux changements d'etat
DashboardsVisualisations

Metriques Standard vs Custom

Standard (5 min) : CPU, Network, DiskOps - gratuites.

Custom (agent) : RAM, espace disque, connexions. Necessitent l'agent CloudWatch.

Important : RAM et disque = PAS standard !

Points Cles Enrichis
⭐
RAM et disque = metriques CUSTOM
CloudWatch par defaut ne voit PAS la memoire RAM ni l'espace disque. Agent CloudWatch necessaire.
⭐
Standard = 5 min | Detailed = 1 min (payant)
Par defaut metriques toutes les 5 min. Detailed monitoring active la collecte toutes les minutes.
⭐
Alarme -> SNS + Auto Scaling + EC2 action
3 etats : OK, ALARM, INSUFFICIENT_DATA. Peut envoyer notification, scaler, ou arreter/redemarrer.
⭐
Logs : Log Groups -> Log Streams -> Events
Hierarchie de logs. Retention configurable de 1 jour a indefini. Necessite agent sur l'instance.

AWS CloudTrail

Journal d'audit AWS

Enregistre TOUTES les actions (appels API). Repond a : QUI a fait QUOI, QUAND, DEPUIS OU ?

CloudWatch vs CloudTrail

CloudWatchCloudTrail
QuestionComment ca PERFORME ?QUI a fait QUOI ?

ExempleCPU 95%User X deleted i-123

Types d'evenements

Management Events : Operations sur ressources (actives par defaut). Data Events : Operations sur donnees (non actives - volume eleve). Insights : Detection d'anomalies.

Points Cles Enrichis
⭐
CloudTrail = audit | CloudWatch = monitoring
"Qui a fait quoi" vs "Comment ca va". Ne les confondez JAMAIS a l'examen.

⭐
Management events = par defaut | Data events = non
Data events (GetObject S3, Invoke Lambda) generent un volume enorme, d'ou leur desactivation par defaut.
⭐
Trail multi-region recommande
Capture les evenements de TOUTES les regions. Indispensable pour une vue complete de la securite.

Athena Integration

SQL serverless sur S3

Requetes SQL directement sur des fichiers dans S3. Pas de serveur. Paiement par quantite de donnees scannees (~5$/To).

Integration CloudTrail + Athena

CloudTrail stocke dans S3. Athena interroge avec SQL pour trouver : qui a modifie quoi, patterns d'acces, anomalies. Aussi pour VPC Flow Logs, ALB Logs.

Points Cles Enrichis
⭐
Athena = serverless, paiement au scan
~5$/To scanne. Pas d'infrastructure. Ideal pour analyses ad-hoc et ponctuelles.
⭐
Formats colonne = economies majeures
Parquet/ORC : ne scanne que les colonnes utiles. Reduit les couts de 30-90% vs CSV/JSON.

Architectures de reference

🏗 Architecture 3 Tiers VPC

Tier 1 - Public (DMZ) : ALB/NLB recoit le trafic via IGW. Bastion Host pour acces SSH admin.
Tier 2 - Prive App : EC2/ECS/Lambda, accessible uniquement via ALB. NAT GW pour updates/patches.
Tier 3 - Prive Data : RDS (Multi-AZ), ElastiCache, accessible uniquement depuis Tier 2.

Regles cles :
- Chaque tier n'accepte le trafic QUE du tier precedent (SG references)
- Sous-reseaux dans 2+ AZ pour la haute disponibilite
- VPC Flow Logs actives sur tous les sous-reseaux
- S3 Gateway Endpoint pour eviter le trafic Internet vers S3

🌍 Architecture Multi-Region DR

Region primaire : VPC + infrastructure complete (EC2, RDS, ALB)
Region secondaire : Infrastructure stand-by ou pilot light

Strategies DR (RTO croissant) :
1. Multi-Site Active/Active - RTO ~0, cout tres eleve
2. Warm Standby - Infrastructure reduite active, RTO minutes
3. Pilot Light - Services core uniquement (RDS replica), RTO dizaines de min
4. Backup & Restore - Snapshots + AMI cross-region, RTO heures

Services cles : Route 53 failover, S3 CRR, RDS Read Replica cross-region, AMI copy

🔐 Architecture Zero Trust VPC

Principes :
- Jamais faire confiance, toujours verifier
- Moindre privilege sur tous les SG/NACL/IAM
- Micro-segmentation : 1 SG par role applicatif

Implementation :
1. SG referencant d'autres SG (pas de CIDR larges)
2. NACLs avec deny explicites sur IP malveillantes
3. VPC Endpoints pour eviter le transit Internet
4. PrivateLink pour les communications inter-services
5. Flow Logs + GuardDuty pour la detection
6. AWS Network Firewall pour inspection deep packet

Formules et calculs

🧮 Calculs CIDR essentiels

Nombre d'IP = 2^(32 - prefix)

/16 = 2^16 = 65 536 IP (max VPC)
/20 = 2^12 = 4 096 IP
/24 = 2^8 = 256 IP (IP disponibles : 256 - 5 = 251)
/28 = 2^4 = 16 IP (min sous-reseau, IP disponibles : 11)

Calcul rapide : Pour un /24, chaque bit de difference = x2 ou /2
/23 = 512 IP, /25 = 128 IP, /26 = 64 IP, /27 = 32 IP

Regle AWS : Toujours soustraire 5 IP reservees par sous-reseau
Attention : Les CIDR de VPC ne doivent JAMAIS se chevaucher pour le peering

💰 Estimation des couts stockage

Cout mensuel = Volume (Go) x Prix/Go/mois

EBS gp3 : 1 To = 1024 x $0.08 = ~$82/mois
EBS io2 : 1 To + 10000 IOPS = ~$145/mois
S3 Standard : 1 To = ~$23/mois
S3 Glacier DA : 1 To = ~$1/mois
EFS Standard : 1 To = ~$300/mois

Optimisations :
- EBS : gp3 > gp2 (20% moins cher, IOPS configurables)
- S3 : Lifecycle Policies pour descendre en classe automatiquement
- EFS : Lifecycle vers IA (jusqu'a 92% d'economie sur fichiers inactifs)
- Snapshots : supprimer les anciens, utiliser DLM (Data Lifecycle Manager)

Scenarios d'examen typiques - Reseau
🎯
Instance privee sans acces Internet ?
-> NAT Gateway dans un sous-reseau PUBLIC + route 0.0.0.0/0 -> NAT GW dans la table de routage privee. Le NAT GW lui-meme a besoin d'une route vers l'IGW.
🎯
Connecter 15 VPC entre eux ?
-> AWS Transit Gateway (hub-and-spoke). Evite n(n-1)/2 = 105 peerings. Supporte aussi les connexions VPN et Direct Connect.
🎯
Acceder a S3 depuis un sous-reseau prive sans Internet ?
-> S3 Gateway Endpoint (gratuit). Ajoute une route dans la table de routage. Trafic reste dans le reseau AWS.
🎯
Connexion hybride fiable et a haute bande passante ?
-> AWS Direct Connect (1-100 Gbps). Pour chiffrement : ajouter VPN par-dessus. Pour redondance : 2 connexions DX dans 2 locations.
🎯
Bloquer une IP specifique qui attaque ?
-> NACL avec une regle DENY sur cette IP (numero de regle bas pour priorite). Les SG ne supportent PAS les deny.
🎯
Resoudre le DNS d'un VPC peere ?
-> Activer DNS resolution dans le peering + Route 53 Resolver (inbound/outbound endpoints) pour les queries cross-VPC.
Scenarios d'examen typiques - Stockage
🎯
Fichiers partages entre 10 serveurs Linux ?
-> Amazon EFS (NFS, multi-instance, multi-AZ, Linux). Mode General Purpose pour latence, Max I/O pour parallelisme massif.
🎯
Logs conformite 7 ans au moindre cout ?
-> S3 Glacier Deep Archive + Vault Lock (WORM). Lifecycle Policy : Standard -> IA (30j) -> Glacier (90j) -> Deep Archive (365j).
🎯
Base de donnees avec 100 000+ IOPS ?
-> EBS io2 Block Express (jusqu'a 256 000 IOPS, 64 TiB). Alternative extreme : Instance Store (latence microseconde).
🎯
Transferer 80 To sans saturer le reseau ?
-> AWS Snowball Edge (80 To). Regle : si transfert reseau > 1 semaine, utiliser Snow Family. 100+ Po = Snowmobile.
🎯
Volume EBS a deplacer vers une autre AZ ?
-> Snapshot EBS (stocke dans S3, niveau Region) -> Creer nouveau volume depuis le snapshot dans la nouvelle AZ.
🎯
Partager des fichiers entre instances Windows ?
-> FSx for Windows File Server (protocole SMB). Supporte Active Directory, DFS, quotas. EFS = Linux only (NFS).
🎯
Acces aleatoire et imprevisible a S3 ?
-> S3 Intelligent-Tiering. Deplace automatiquement entre tiers (frequent, infrequent, archive) sans frais de recuperation.

🎯

Scenarios d'examen typiques - Haute Disponibilite
🎯
Application web hautement disponible ?
-> ALB + Auto Scaling Group (min 2 instances) + Multi-AZ. RDS Multi-AZ pour la BDD. Route 53 health checks.
🎯
Failover automatique en cas de panne AZ ?
-> Deployer dans 2+ AZ. ALB distribue le trafic. ASG remplace les instances defaillantes. RDS failover automatique (Multi-AZ).

Pieges frequents - Reseau et Securite
⚠
SG vs NACL - Le piege #1 de l'examen
SG = stateful (pas de regle retour necessaire) / NACL = stateless (regles IN et OUT obligatoires + ports ephemeres 1024-65535). SG = ALLOW only / NACL = ALLOW + DENY.
⚠
NAT Gateway = sous-reseau PUBLIC obligatoirement
Jamais dans le prive ! Il a besoin d'une route vers l'IGW pour fonctionner. Route dans le sous-reseau PRIVE : 0.0.0.0/0 -> NAT GW.
⚠
VPC Peering = NON transitif
A-B et B-C ne donne PAS A-C. Chaque paire necessite son propre peering + routes dans les DEUX tables de routage. CIDR ne doivent PAS se chevaucher.
⚠
Direct Connect ≠ chiffre par defaut
C'est une connexion PRIVEE mais pas chiffree. Pour le chiffrement, ajouter un VPN IPsec par-dessus la connexion DX.
⚠
Gateway Endpoint vs Interface Endpoint
Gateway = S3 et DynamoDB UNIQUEMENT, GRATUIT, modifie la route table. Interface = tous les autres services, PAYANT, cree une ENI dans votre sous-reseau.
⚠
SG par defaut du VPC par defaut
Autorise TOUT le trafic sortant et tout le trafic entrant DEPUIS le meme SG. Pas depuis l'exterieur ! Un SG custom refuse tout par defaut (in et out).
Pieges frequents - Stockage
⚠
EBS vs EFS - Ne JAMAIS confondre
EBS = 1 instance, 1 AZ, block storage / EFS = multi-instances, multi-AZ, LINUX ONLY (NFS). EFS est elastique (pas de provisionnement), EBS a une taille fixe.
⚠
S3 n'est PAS un filesystem
Stockage OBJET via API HTTP. On ne peut PAS le monter sur une instance comme un disque. Pour un FS : EBS, EFS ou FSx.
⚠
Instance Store = EPHEMERE (donnees perdues !)
Perdu au STOP, TERMINATE, ou defaillance hardware. Survit UNIQUEMENT au reboot. Jamais pour donnees critiques.
⚠
HDD (st1/sc1) ne peuvent PAS etre boot volumes
Seuls les SSD (gp2, gp3, io1, io2) peuvent etre des volumes de demarrage. st1/sc1 = donnees uniquement.
⚠
EBS Multi-Attach = io1/io2 UNIQUEMENT + meme AZ
Pas de multi-attach sur gp2/gp3. Les instances DOIVENT etre dans la meme AZ. Max 16 instances. Application doit gerer la concurrence.
⚠
Versioning S3 : suspendu ≠ desactive
Une fois active, le versioning ne peut JAMAIS etre completement desactive. Seulement suspendu. Les versions existantes restent.
⚠
Suppression anticipee = frais !
Glacier Flexible : min 90 jours. Deep Archive : min 180 jours. Supprimer avant = payer le reste de la periode minimum.
⚠
DeleteOnTermination = true par defaut (root volume)
Le volume root EBS est SUPPRIME quand l'instance est terminee. Mettre a false pour conserver. Les volumes additionnels = false par defaut.
Pieges frequents - Surveillance
⚠
CloudWatch vs CloudTrail - Ne JAMAIS confondre
CloudWatch = PERFORMANCES (metriques, logs, alarmes). CloudTrail = AUDIT (qui a fait quoi, quand, depuis ou). 2 services completement differents.
⚠
RAM = PAS une metrique standard CloudWatch
L'hyperviseur ne voit pas la memoire de l'instance. Necessite l'agent CloudWatch installe DANS l'instance. Piege classique #1.
⚠
CloudTrail Data Events = desactives par defaut
GetObject S3, Invoke Lambda = Data Events, non loggues par defaut (trop de volume). A activer explicitement dans le Trail.
⚠
VPC Flow Logs = metadonnees UNIQUEMENT
Capturent IP source/dest, port, protocole, ACCEPT/REJECT. JAMAIS le contenu des paquets. Pour le contenu : inspection deep packet.
⚠
Detailed Monitoring = 1 min (PAYANT)
Par defaut : Basic Monitoring = 5 min (gratuit). Detailed = 1 min mais payant. A activer explicitement sur chaque instance.
⚠
Athena = pay per scan, pas per query
Facture au volume de donnees SCANNEES (~$5/To). Utiliser Parquet/ORC (colonne) pour reduire 30-90%. Partitionner les donnees !
Mnemoniques pour l'examen
💡
NACL = "Network Access Control List"
Numero de regle (ordre), Allow + Deny, Couche sous-reseau, Less context (stateless). Retenir : "NACL = Numbered, Allow/Deny, subnet-level, stateLess"
💡
5 IP reservees = "RRDRF"
Reseau (.0), Routeur (.1), DNS (.2), Reserve (.3), Fin/broadcast (derniere). Mnemonique : "RRouDReF" = 5 IP perdues par sous-reseau.
💡
S3 Durabilite = "11 neufs"
99.999999999% = si vous stockez 10 millions d'objets, vous pouvez en perdre 1 tous les 10 000 ans. Retenir : "S3 = quasi-indestructible".
💡
Snow Family = "CSE" par taille
Cone = 8 To, Snowball = 80 To, Snowball Edge = 80 To + compute. Snowmobile = 100 Po (le camion). Regle : > 1 semaine de transfert = Snow.
💡
Chiffrement EBS = "AAAA"
At rest (AES-256), AWS KMS, Automatique en transit (EBS<->EC2), Applique aux snapshots aussi. Un volume chiffre = tout chiffre.
