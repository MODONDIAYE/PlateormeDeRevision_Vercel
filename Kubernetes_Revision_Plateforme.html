1 - Problematique : Pourquoi orchestrer des conteneurs ?

0/5

Le probleme des conteneurs isoles : Docker permet de creer des conteneurs, mais en production, une application moderne peut necessiter des dizaines voire des centaines de conteneurs. Gerer manuellement le lancement, l'arret, la mise a jour et la surveillance de chaque conteneur devient rapidement impossible. Imaginez un chef d'orchestre qui devrait diriger 100 musiciens un par un, sans partition commune.

La mise a l'echelle (scaling) : Quand le trafic augmente (par exemple lors d'un Black Friday), il faut lancer davantage de conteneurs pour absorber la charge. Quand le trafic diminue, il faut en reduire le nombre pour economiser des ressources. Faire cela manuellement 24h/24 est impensable. C'est comme si un restaurateur devait embaucher et licencier des serveurs en temps reel selon le nombre de clients.

La haute disponibilite : Si un serveur tombe en panne, les conteneurs qui y tournaient doivent etre automatiquement relances sur un autre serveur, sans interruption de service pour l'utilisateur. Sans orchestrateur, un administrateur devrait etre de garde en permanence pour reagir aux pannes.

La complexite du deploiement : Mettre a jour une application sans temps d'arret (rolling update) necessite de remplacer progressivement les anciens conteneurs par les nouveaux. Si une erreur survient, il faut pouvoir revenir en arriere instantanement (rollback). Gerer cela a la main est source d'erreurs et de stress.

L'orchestration comme solution : Un orchestrateur de conteneurs est un logiciel qui automatise tout cela : deploiement, mise a l'echelle, repartition de charge, auto-reparation, mises a jour progressives et rollback. Kubernetes est devenu le standard mondial pour cette tache. Il transforme un ensemble de serveurs en une plateforme unifiee ou deployer des applications.

  • Conteneurs isoles = ingerable en production a grande echelle. L'orchestration automatise la gestion.
  • Scaling automatique = adapter le nombre de conteneurs au trafic.
  • Haute disponibilite = relancer automatiquement les conteneurs en cas de panne.
  • Rolling update = mise a jour sans interruption de service.
  • Rollback = retour instantane a la version precedente en cas de probleme.
  • Kubernetes = orchestrateur standard de l'industrie.

2 - Presentation de Kubernetes

0/5

Origine et histoire : Kubernetes a ete cree par Google en 2014 et est inspire de Borg, le systeme interne de Google qui gere des milliards de conteneurs chaque semaine. Google a rendu Kubernetes open source pour partager 15 ans d'experience en orchestration de conteneurs avec la communaute mondiale.

Qu'est-ce que Kubernetes : Kubernetes (souvent abrege K8s — le "8" represente les 8 lettres entre le K et le S) est une plateforme open source d'orchestration de conteneurs. Elle automatise le deploiement, la mise a l'echelle et la gestion des applications conteneurisees sur un cluster de machines.

La CNCF (Cloud Native Computing Foundation) : Depuis 2015, Kubernetes est heberge par la CNCF, une fondation sous la Linux Foundation. La CNCF regroupe des geants comme Google, Microsoft, Amazon, Red Hat et assure la gouvernance neutre du projet. Kubernetes a ete le premier projet "graduate" de la CNCF.

Pourquoi K8s domine : Kubernetes est devenu le standard de facto pour l'orchestration car il est portable (fonctionne sur n'importe quel cloud ou on-premise), extensible (architecture par plugins), a une enorme communaute (plus de 3000 contributeurs), et est supporte par tous les grands fournisseurs cloud (AWS EKS, Google GKE, Azure AKS).

Ce que Kubernetes n'est PAS : Kubernetes n'est pas un PaaS (Platform as a Service) complet. Il ne fournit pas de base de donnees integree, de middleware ou de pipeline CI/CD. Il ne construit pas les images Docker non plus. C'est une couche d'infrastructure qui orchestre les conteneurs, sur laquelle on bâtit le reste.

  • Cree par Google en 2014, inspire de Borg (15 ans d'experience).
  • K8s = abreviation (K + 8 lettres + s).
  • Heberge par la CNCF (Cloud Native Computing Foundation) depuis 2015.
  • Standard de facto : portable, extensible, enorme communaute.
  • Supporte par AWS (EKS), Google (GKE), Azure (AKS).
  • N'est PAS un PaaS complet : pas de CI/CD, pas de DB integree.

3 - Concepts fondamentaux de Kubernetes

0/6

Pod : Le Pod est la plus petite unite deployable dans Kubernetes. Il contient un ou plusieurs conteneurs qui partagent le meme reseau et le meme stockage. Pensez a un Pod comme un appartement : les colocataires (conteneurs) partagent la meme adresse et les memes pieces communes. En general, on met un seul conteneur par Pod.

Node : Un Node est une machine physique ou virtuelle dans le cluster Kubernetes. Il execute les Pods. Il existe deux types de Nodes : les Master Nodes (qui gerent le cluster) et les Worker Nodes (qui executent les applications). C'est comme les travailleurs d'une usine : chacun execute une partie du travail.

ReplicaSet : Un ReplicaSet garantit qu'un nombre specifie de replicas (copies) d'un Pod fonctionne a tout moment. Si un Pod meurt, le ReplicaSet en cree un nouveau automatiquement. C'est comme un manager qui s'assure qu'il y a toujours exactement 3 caissiers a la caisse.

Deployment : Un Deployment est un objet de haut niveau qui gere les ReplicaSets et permet les mises a jour declaratives. On declare l'etat desire (ex: 3 replicas de mon app en version 2.0) et Kubernetes se charge d'y arriver. C'est la maniere recommandee de deployer des applications stateless.

Service : Un Service est une abstraction qui expose un ensemble de Pods sous une adresse IP stable et un nom DNS. Les Pods sont ephemeres (ils peuvent mourir et renaitre avec une nouvelle IP), mais le Service fournit un point d'acces permanent. C'est comme un standard telephonique : le numero ne change pas meme si les operateurs changent.

Ingress et Gateway API : L'Ingress gere l'acces HTTP/HTTPS externe au cluster en definissant des regles de routage (ex: /api va au service A, /web va au service B). La Gateway API est le successeur plus moderne et plus flexible de l'Ingress, avec une meilleure separation des responsabilites entre les equipes plateforme et developpement.

ConfigMap et Secret : Un ConfigMap stocke des donnees de configuration (variables d'environnement, fichiers de config) de maniere non confidentielle. Un Secret stocke des donnees sensibles (mots de passe, tokens, cles) encodees en base64. Les deux permettent de separer la configuration du code applicatif.

Volumes et StatefulSet : Les Volumes permettent de stocker des donnees de maniere persistante, independamment du cycle de vie des Pods (un PersistentVolume survit a la destruction d'un Pod). Un StatefulSet est concu pour les applications avec etat (bases de donnees) : il garantit un identifiant stable, un stockage persistant et un ordre de deploiement.

Namespace, DaemonSet, Job/CronJob : Un Namespace isole logiquement les ressources dans un cluster (comme des dossiers). Un DaemonSet garantit qu'un Pod tourne sur CHAQUE Node (ex: agent de monitoring). Un Job execute une tache ponctuelle jusqu'a sa completion, et un CronJob la planifie selon un horaire (comme un cron Unix).

  • Pod = plus petite unite, 1+ conteneurs partageant reseau/stockage.
  • Node = machine qui execute les Pods (Master ou Worker).
  • ReplicaSet = maintient N copies d'un Pod en vie.
  • Deployment = gere ReplicaSets + mises a jour declaratives.
  • Service = IP stable pour acceder aux Pods ephemeres.
  • Ingress/Gateway API = routage HTTP/HTTPS externe.
  • ConfigMap (config) / Secret (donnees sensibles).
  • StatefulSet = apps avec etat (DB), stockage persistant.
  • Namespace = isolation logique, DaemonSet = 1 Pod/Node, Job = tache ponctuelle.

4 - Architecture de Kubernetes

0/5

Le Control Plane (Plan de controle) : C'est le "cerveau" du cluster Kubernetes. Il prend toutes les decisions globales : sur quel Node placer un Pod, comment reagir a une panne, quand scaler. Il est generalement deploye sur des Master Nodes separes des Worker Nodes pour la fiabilite.

API Server (kube-apiserver) : C'est la porte d'entree unique du cluster. Toute commande kubectl, toute requete interne passe par l'API Server. Il valide les requetes, les authentifie et les enregistre dans etcd. C'est le standard telephonique central : rien ne se fait sans passer par lui.

etcd : C'est la base de donnees cle-valeur distribuee qui stocke TOUT l'etat du cluster (quels Pods existent, sur quels Nodes, quelles configurations). Si etcd est perdu, le cluster est perdu. C'est la memoire du cluster, comme un registre officiel qui contient toutes les informations vitales.

Scheduler (kube-scheduler) : Quand un nouveau Pod doit etre cree, le Scheduler decide sur quel Node le placer. Il prend en compte les ressources disponibles (CPU, RAM), les contraintes (affinites, tolerations) et l'equilibrage de charge. C'est comme un directeur d'hotel qui assigne les chambres aux clients selon les disponibilites.

Controller Manager (kube-controller-manager) : Il execute les boucles de controle qui surveillent l'etat du cluster et agissent pour le ramener vers l'etat desire. Par exemple, le ReplicaSet Controller verifie que le bon nombre de Pods existe, le Node Controller detecte les pannes de Nodes. C'est un ensemble de "gardiens" automatiques.

kubelet : C'est l'agent qui tourne sur CHAQUE Worker Node. Il recoit les instructions de l'API Server et s'assure que les conteneurs d'un Pod sont en cours d'execution et en bonne sante. C'est le contremaître sur le terrain qui execute les ordres du management.

kube-proxy : C'est le composant reseau qui tourne sur chaque Node. Il maintient les regles reseau (iptables ou IPVS) qui permettent la communication vers les Pods via les Services. Quand un client accede a un Service, kube-proxy redirige le trafic vers le bon Pod.

Container Runtime : C'est le logiciel responsable de l'execution effective des conteneurs. Kubernetes supporte plusieurs runtimes via l'interface CRI (Container Runtime Interface) : containerd (le plus repandu), CRI-O, et d'autres. Docker n'est plus supporte directement depuis Kubernetes 1.24.

  • Control Plane = cerveau du cluster (decisions globales).
  • API Server = porte d'entree unique, valide et route tout.
  • etcd = base cle-valeur qui stocke TOUT l'etat du cluster.
  • Scheduler = decide sur quel Node placer chaque Pod.
  • Controller Manager = boucles de controle (etat desire vs actuel).
  • kubelet = agent sur chaque Worker, execute les Pods.
  • kube-proxy = regles reseau pour les Services sur chaque Node.
  • Container Runtime = containerd (CRI), Docker retire depuis v1.24.

5 - Communication et Reseau dans Kubernetes

0/5

ClusterIP : C'est le type de Service par defaut. Il attribue une adresse IP interne au cluster, accessible uniquement depuis l'interieur. Les autres Pods du cluster peuvent joindre ce Service via son ClusterIP ou son nom DNS. C'est comme un numero de telephone interne dans une entreprise : il ne fonctionne pas depuis l'exterieur.

NodePort : Ce type de Service ouvre un port fixe (entre 30000 et 32767) sur CHAQUE Node du cluster. Le trafic arrivant sur ce port est redirige vers le Service. C'est simple mais limite : on expose directement les Nodes, ce qui n'est pas ideal en production. C'est utile pour le developpement ou les tests.

LoadBalancer : Ce type de Service provisionne automatiquement un repartiteur de charge (load balancer) externe aupres du fournisseur cloud (AWS ELB, Azure LB, GCP LB). Le trafic arrive sur le LB qui le distribue aux Nodes. C'est la methode standard pour exposer un Service en production sur le cloud.

Ingress : Un Ingress est un objet qui definit des regles de routage HTTP/HTTPS vers les Services. Un Ingress Controller (comme Nginx Ingress ou Traefik) implemente ces regles. L'avantage est de pouvoir router plusieurs domaines/chemins vers differents Services avec un seul point d'entree et un seul certificat TLS.

CoreDNS : CoreDNS est le serveur DNS interne du cluster Kubernetes. Il permet aux Pods de se trouver par nom. Par exemple, un Pod peut acceder au Service "mon-api" simplement en utilisant l'URL http://mon-api.default.svc.cluster.local. Chaque Service recoit automatiquement une entree DNS.

Network Policies : Les Network Policies sont des regles de pare-feu au niveau du Pod. Par defaut, tous les Pods peuvent communiquer entre eux. Les Network Policies permettent de restreindre le trafic entrant (ingress) et sortant (egress) de certains Pods. C'est essentiel pour la securite : par exemple, seul le backend peut acceder a la base de donnees.

  • ClusterIP = IP interne uniquement, type par defaut.
  • NodePort = port fixe (30000-32767) sur chaque Node, pour dev/tests.
  • LoadBalancer = LB externe du cloud, standard en production.
  • Ingress = routage HTTP/HTTPS, plusieurs domaines/chemins vers un point d'entree.
  • CoreDNS = resolution de noms interne (service.namespace.svc.cluster.local).
  • Network Policies = pare-feu au niveau Pod, restreindre les communications.

6 - Installation de Kubernetes

0/5

Minikube : Minikube est un outil qui permet de lancer un cluster Kubernetes local sur une seule machine (dans une VM ou un conteneur Docker). C'est ideal pour l'apprentissage et le developpement. Il cree un cluster a un seul Node avec tout le necessaire. C'est comme avoir un mini-laboratoire personnel pour s'exercer sans risque.

kubeadm : kubeadm est l'outil officiel pour installer un cluster Kubernetes "from scratch" sur des serveurs existants. Il gere l'initialisation du Control Plane (kubeadm init) et l'ajout de Worker Nodes (kubeadm join). C'est la methode standard pour les environnements on-premise ou les VMs en production. Il demande plus de configuration manuelle mais offre un controle total.

k3s : k3s est une distribution Kubernetes ultra-legere creee par Rancher (SUSE). Elle est conçue pour l'IoT, l'edge computing et les environnements a ressources limitees. Toute l'installation tient dans un seul binaire de moins de 100 Mo. Elle supprime les composants non essentiels et utilise SQLite au lieu d'etcd par defaut.

Services manages (EKS, GKE, AKS) : Les grands fournisseurs cloud proposent des clusters Kubernetes geres : Amazon EKS (Elastic Kubernetes Service), Google GKE (Google Kubernetes Engine), Azure AKS (Azure Kubernetes Service). Le Control Plane est gere par le cloud provider (maintenance, mises a jour, haute disponibilite), l'utilisateur ne gere que ses Worker Nodes et ses applications.

kubectl : kubectl est l'outil en ligne de commande (CLI) pour interagir avec un cluster Kubernetes. Il communique avec l'API Server et permet de deployer des applications, inspecter des ressources, consulter des logs et debugger. C'est l'equivalent d'un tableau de bord en ligne de commande, indispensable au quotidien pour tout operateur Kubernetes.

  • Minikube = cluster local 1 Node pour dev/apprentissage.
  • kubeadm = installation officielle from scratch (on-premise).
  • k3s = distribution ultra-legere (IoT, edge, 100 Mo).
  • EKS (AWS), GKE (Google), AKS (Azure) = clusters manages.
  • kubectl = CLI indispensable pour piloter le cluster.
  • Managed = Control Plane gere par le cloud, vous gerez les apps.

7 - Commandes kubectl de base

0/5

kubectl get : Cette commande liste les ressources d'un type donne. Par exemple, "kubectl get pods" affiche tous les Pods du namespace courant, "kubectl get nodes" liste les Nodes du cluster, "kubectl get services" liste les Services. L'option "-o wide" ajoute des colonnes supplementaires (IP, Node). C'est la commande la plus utilisee pour avoir une vue d'ensemble.

kubectl describe : Cette commande affiche des informations detaillees sur une ressource specifique. "kubectl describe pod mon-pod" montre les events, l'etat des conteneurs, les volumes montes, les conditions, etc. C'est essentiel pour le debug : si un Pod ne demarre pas, describe montre pourquoi (image introuvable, ressources insuffisantes, erreur de permission).

kubectl apply : Cette commande applique une configuration declarative depuis un fichier YAML. "kubectl apply -f deployment.yaml" cree ou met a jour les ressources decrites dans le fichier. C'est la methode declarative recommandee : on declare l'etat desire et Kubernetes se charge de l'atteindre. Si la ressource existe deja, elle est mise a jour.

kubectl delete : Cette commande supprime des ressources. "kubectl delete pod mon-pod" supprime un Pod, "kubectl delete -f fichier.yaml" supprime toutes les ressources definies dans le fichier. Attention : supprimer un Deployment supprime aussi ses Pods. La suppression est definitive, il n'y a pas de corbeille.

kubectl logs et exec : "kubectl logs mon-pod" affiche les logs (stdout/stderr) d'un conteneur dans un Pod. L'option "-f" suit les logs en temps reel (comme tail -f). "kubectl exec -it mon-pod -- /bin/bash" ouvre un terminal interactif dans le conteneur, ce qui est tres utile pour le debug en direct.

kubectl scale et rollout : "kubectl scale deployment mon-app --replicas=5" change le nombre de replicas a la volee. "kubectl rollout status deployment/mon-app" verifie l'avancement d'une mise a jour. "kubectl rollout undo deployment/mon-app" effectue un rollback vers la version precedente. Ces commandes sont critiques pour la gestion operationnelle.

  • kubectl get = lister les ressources (pods, nodes, services).
  • kubectl describe = details + events pour le debug.
  • kubectl apply -f = deploiement declaratif via YAML.
  • kubectl delete = supprimer des ressources (irreversible).
  • kubectl logs -f = logs en temps reel d'un Pod.
  • kubectl exec -it = terminal interactif dans un conteneur.
  • kubectl scale = ajuster le nombre de replicas.
  • kubectl rollout undo = rollback vers version precedente.

8 - Projet fil rouge : Deployer une application complete

0/5

Deployer une application : Pour deployer une application sur Kubernetes, on cree un fichier YAML definissant un Deployment. Ce fichier specifie le nom de l'image Docker, le nombre de replicas, les ports, les limites de ressources (CPU/RAM) et les variables d'environnement. La commande "kubectl apply -f deployment.yaml" envoie cette declaration au cluster qui cree les Pods correspondants.

Exposer avec un Service : Une fois les Pods crees, il faut les exposer via un Service. Le Service de type ClusterIP rend l'application accessible a l'interieur du cluster. Pour un acces externe, on utilise un Service LoadBalancer ou un Ingress. Le Service selectionne les Pods via des labels (ex: app=mon-api) et repartit la charge entre eux.

Configurer avec ConfigMaps et Ingress : Les ConfigMaps permettent de passer des variables d'environnement sans les coder en dur dans l'image. L'Ingress definit les regles de routage HTTP : par exemple, app.example.com/api pointe vers le Service backend et app.example.com pointe vers le Service frontend. Un Ingress Controller (Nginx) implemente ces regles.

Horizontal Pod Autoscaler (HPA) : Le HPA surveille les metriques (CPU, memoire, metriques custom) et ajuste automatiquement le nombre de replicas. Par exemple, "si l'utilisation CPU depasse 70%, ajouter des Pods jusqu'a un maximum de 10". Quand la charge diminue, le HPA reduit le nombre de Pods. C'est le scaling automatique en action.

Bonnes pratiques de deploiement : Toujours definir des limites de ressources (requests et limits) pour eviter qu'un Pod consomme toutes les ressources du Node. Utiliser des health checks (readinessProbe et livenessProbe) pour que Kubernetes sache si le Pod est pret a recevoir du trafic ou doit etre redemarre. Versionner les images (jamais "latest" en production).

  • Deployment YAML = image + replicas + ports + resources.
  • Service = expose les Pods via labels, repartit la charge.
  • ConfigMap = configuration externalisee (variables d'env).
  • Ingress + Controller = routage HTTP/HTTPS avec regles.
  • HPA = scaling automatique base sur CPU/memoire.
  • Bonnes pratiques : limits/requests, probes, pas de "latest".

9 - References et Ecosysteme Kubernetes

0/5

Helm : Helm est le gestionnaire de paquets de Kubernetes, souvent appele le "apt/yum de Kubernetes". Il permet d'installer des applications complexes (ex: Prometheus, Grafana, Nginx) en une seule commande via des "charts" (paquets preconfigures). Un chart Helm regroupe tous les fichiers YAML necessaires avec des valeurs parametrables.

Prometheus et Grafana : Prometheus est l'outil de monitoring standard de l'ecosysteme Kubernetes (projet CNCF). Il collecte les metriques des Pods et des Nodes. Grafana est l'outil de visualisation qui cree des tableaux de bord a partir des donnees Prometheus. Ensemble, ils forment la stack de monitoring de reference pour Kubernetes.

Istio et les Service Meshes : Un Service Mesh est une couche d'infrastructure qui gere la communication entre les microservices (chiffrement mTLS, observabilite, routage avance). Istio est le Service Mesh le plus connu pour Kubernetes. Il ajoute un proxy sidecar (Envoy) a chaque Pod pour intercepter et gerer le trafic reseau.

GitOps et ArgoCD : GitOps est une pratique ou l'etat desire du cluster est stocke dans un depot Git. Un outil comme ArgoCD surveille le depot et synchronise automatiquement le cluster avec le contenu du Git. Tout changement passe par un commit Git, ce qui offre un historique complet, la revue de code et le rollback facile.

Certifications Kubernetes : La CNCF propose des certifications officielles : CKA (Certified Kubernetes Administrator) pour l'administration de clusters, CKAD (Certified Kubernetes Application Developer) pour le developpement d'applications sur Kubernetes, et CKS (Certified Kubernetes Security Specialist) pour la securite. Ce sont des examens pratiques tres respectes dans l'industrie.

  • Helm = gestionnaire de paquets, "charts" pour deployer des apps complexes.
  • Prometheus + Grafana = monitoring + visualisation standard.
  • Istio = Service Mesh (mTLS, observabilite, proxy Envoy).
  • GitOps + ArgoCD = etat du cluster versionne dans Git.
  • CKA / CKAD / CKS = certifications officielles CNCF.

Types de Services Kubernetes

  • ClusterIP - Accessible uniquement depuis l'interieur du cluster (type par defaut)
  • NodePort - Expose un port (30000-32767) sur chaque Node du cluster
  • LoadBalancer - Provisionne un LB externe (cloud), standard en production
  • Ingress - Routage HTTP/HTTPS avec regles (domaines, chemins, TLS)

Objets Kubernetes principaux

  • Pod = plus petite unite, 1+ conteneurs
  • ReplicaSet = garantit N replicas d'un Pod
  • Deployment = gere ReplicaSets + rolling updates
  • Service = IP stable + load balancing
  • Ingress = routage HTTP/HTTPS externe
  • ConfigMap / Secret = configuration externalisee
  • PersistentVolume = stockage persistant
  • StatefulSet = apps avec etat (DB)
  • DaemonSet = 1 Pod par Node
  • Job / CronJob = taches ponctuelles / planifiees

Commandes essentielles

# Deployer une application kubectl apply -f deployment.yaml # Verifier l'etat des pods kubectl get pods -o wide # Diagnostiquer un pod en erreur kubectl describe pod mon-pod kubectl logs mon-pod # Scaler un deployment kubectl scale deployment mon-app --replicas=5 # Mise a jour rolling kubectl set image deployment/mon-app app=mon-image:v2.0 # Rollback en cas d'erreur kubectl rollout undo deployment/mon-app # Port-forward pour tester localement kubectl port-forward svc/mon-service 8080:80 # Autoscaling kubectl autoscale deployment mon-app --min=2 --max=10 --cpu-percent=70
