1 - Problematique du Monitoring

0/5

Pourquoi le monitoring est-il indispensable ? Imaginez une application en production sans aucune visibilite : vous ne savez pas si elle fonctionne correctement, si la charge augmente dangereusement, ou si un disque est presque plein. Sans monitoring, vous decouvrez les problemes APRES que les utilisateurs les signalent — souvent trop tard. Le monitoring transforme une approche reactive ("on repare quand ca casse") en approche proactive ("on anticipe avant que ca casse").

Consequences de l'absence d'observabilite : Sans observabilite, les equipes naviguent a l'aveugle. Les pannes durent plus longtemps (MTTR eleve), les causes racines sont difficiles a identifier, la scalabilite est impossible a planifier, et les SLA sont violes frequemment. Chaque minute d'indisponibilite coute de l'argent (pertes de revenus, reputation degradee, penalites contractuelles). Amazon estime qu'une minute de panne lui coute environ 220 000 dollars.

MTTR (Mean Time To Recovery) : Le temps moyen de recuperation apres une panne. C'est un indicateur critique : plus il est bas, meilleure est votre capacite de reaction. Le monitoring reduit drastiquement le MTTR car il permet de detecter les problemes immediatement (alertes), de diagnostiquer rapidement (metriques, dashboards) et de verifier la resolution (retour a la normale). Un bon MTTR en production est inferieur a 30 minutes.

SLA, SLO et SLI — Les trois piliers de la fiabilite : Le SLA (Service Level Agreement) est le contrat avec le client ("99.9% de disponibilite/mois"). Le SLO (Service Level Objective) est l'objectif interne plus strict ("99.95%") pour avoir une marge de securite. Le SLI (Service Level Indicator) est la metrique reelle mesuree ("temps de reponse < 200ms pour 99% des requetes"). Sans monitoring, impossible de mesurer les SLI et donc de savoir si on respecte ses SLO/SLA.

Les trois piliers de l'observabilite : 1) Les Metriques (donnees numeriques dans le temps — Prometheus). 2) Les Logs (evenements textuels detailles — Loki, ELK). 3) Les Traces (suivi d'une requete a travers les microservices — Jaeger, Tempo). Ensemble, ils offrent une vision complete du systeme. Prometheus et Grafana couvrent principalement le pilier "Metriques" avec une integration possible aux logs et traces.

  • Sans monitoring = navigation a l'aveugle. Pannes decouvertes par les utilisateurs.
  • MTTR = temps de recuperation. Le monitoring le reduit drastiquement.
  • SLA = contrat client, SLO = objectif interne, SLI = metrique reelle.
  • 3 piliers observabilite : Metriques, Logs, Traces.
  • Proactif vs Reactif. Anticiper plutot que subir.

2 - Presentation Prometheus & Grafana

0/5

Prometheus — Qu'est-ce que c'est ? Prometheus est un systeme de monitoring et d'alerte open source, originalement developpe par SoundCloud en 2012, puis donne a la CNCF (Cloud Native Computing Foundation) en 2016. C'est le deuxieme projet a avoir obtenu le statut "graduated" de la CNCF apres Kubernetes. Il est ecrit en Go et specialise dans la collecte de metriques numeriques sous forme de series temporelles (time series).

Le modele Pull-based : Contrairement a d'autres systemes (comme Graphite ou StatsD qui recoivent des donnees poussees), Prometheus va CHERCHER les metriques aupres des cibles (targets) — c'est le modele "pull". Toutes les X secondes (scrape_interval, par defaut 15s), Prometheus interroge les endpoints /metrics de chaque service. Avantages : pas besoin de configurer chaque service pour pousser, detection automatique des services down (pas de reponse = alerte).

Time Series (series temporelles) : Prometheus stocke toutes les donnees sous forme de series temporelles : une suite de valeurs horodatees (timestamp + valeur). Chaque serie est identifiee par un nom de metrique et un ensemble de labels (cles/valeurs). Exemple : http_requests_total{method="GET", status="200"} = nombre de requetes GET reussies. Cette approche dimensionnelle est extremement flexible pour filtrer et agreger les donnees.

Grafana — Qu'est-ce que c'est ? Grafana est une plateforme de visualisation open source creee en 2014 par Torkel Odegaard. Elle ne collecte PAS de donnees elle-meme — elle se connecte a des sources de donnees (Prometheus, InfluxDB, Elasticsearch, MySQL, etc.) et les affiche sous forme de dashboards interactifs avec des graphiques, jauges, tableaux et alertes visuelles. Grafana est le standard de facto pour la visualisation de metriques.

Historique et alternatives : Avant Prometheus : Nagios (1999, checks actifs, pas de metriques riches), Zabbix (2001, agent-based), Graphite (2008, push-based). Alternatives modernes : InfluxDB + Telegraf (push-based, SQL-like), Datadog (SaaS payant), Victoria Metrics (compatible Prometheus, meilleure retention), Thanos/Cortex (Prometheus multi-cluster). Prometheus + Grafana reste la stack la plus populaire en environnement cloud-native.

  • Prometheus = CNCF graduated, Go, pull-based, time series.
  • Pull = Prometheus va chercher les metriques (scrape).
  • Grafana = visualisation, multi-sources, dashboards.
  • Cree par SoundCloud (2012), CNCF (2016).
  • Alternatives : Datadog (SaaS), Victoria Metrics, Thanos.

3 - Concepts fondamentaux

0/6

Targets et Scraping : Une target est un endpoint HTTP que Prometheus interroge regulierement (scrape). Chaque target expose un endpoint /metrics au format texte specifique. Le scrape_interval (par defaut 15 secondes) definit la frequence de collecte. Si une target ne repond plus, Prometheus le detecte immediatement et peut declencher une alerte "instance down".

Types de metriques — Counter : Un compteur qui ne fait que monter (ou se remettre a zero au redemarrage). Exemples : nombre total de requetes HTTP (http_requests_total), nombre d'erreurs (errors_total). On utilise rate() pour calculer le taux par seconde. JAMAIS utiliser un counter pour une valeur qui peut diminuer (comme la memoire libre).

Types de metriques — Gauge : Une jauge qui peut monter ET descendre. Exemples : temperature CPU, memoire utilisee, nombre de goroutines actives, nombre de connexions en cours. Contrairement au counter, on peut lire directement la valeur d'un gauge sans appliquer rate().

Types de metriques — Histogram : Mesure la distribution de valeurs dans des "buckets" (tranches) predefinis. Exemple : temps de reponse HTTP. Un histogram cree automatiquement 3 series : _bucket (repartition par tranche), _sum (somme totale), _count (nombre d'observations). Permet de calculer des percentiles (p99, p95) avec histogram_quantile().

Types de metriques — Summary : Similaire a l'histogram mais calcule les quantiles COTE CLIENT (dans l'application). Avantage : pas besoin de definir des buckets a l'avance. Inconvenient : les quantiles ne sont PAS agregeables entre instances. Preferer l'histogram dans la majorite des cas.

Exporters : Des programmes qui exposent les metriques d'un systeme au format Prometheus. node_exporter = metriques systeme Linux (CPU, RAM, disque, reseau). blackbox_exporter = sonde HTTP/TCP/ICMP externe (verifier si un site repond). mysqld_exporter, redis_exporter, etc. On peut aussi creer des exporters custom avec les librairies client Prometheus (Go, Python, Java).

AlertManager : Composant separe qui gere les alertes emises par Prometheus. Il s'occupe du routing (quelle alerte va a quelle equipe), du grouping (regrouper les alertes similaires), de l'inhibition (supprimer les alertes redondantes) et du silencing (mettre en sourdine temporairement). Les receivers definissent les canaux : email, Slack, PagerDuty, webhooks.

PromQL (Prometheus Query Language) : Le langage de requete pour interroger les metriques. Exemples : rate(http_requests_total[5m]) = taux de requetes par seconde sur 5 min. sum by(status) (rate(http_requests_total[5m])) = meme chose groupee par code HTTP. C'est un langage puissant qui permet filtrage, agregation, operations mathematiques et fonctions temporelles.

Recording Rules : Des requetes PromQL pre-calculees et sauvegardees comme nouvelles metriques. Utiles pour les requetes complexes utilisees souvent dans les dashboards. Elles reduisent la charge de calcul au moment de l'affichage. Definies dans des fichiers YAML referencies dans prometheus.yml.

Grafana — Sources de donnees et Dashboards : Grafana se connecte a Prometheus (et autres) via des "data sources". Les dashboards sont composes de "panels" (graphiques, jauges, tableaux). Les variables permettent de rendre les dashboards dynamiques (choisir un namespace, une instance). Les annotations marquent des evenements sur les graphiques (deploiements, incidents). Les notifications Grafana permettent aussi de definir des alertes directement dans Grafana.

  • Counter = toujours croissant. Gauge = monte et descend.
  • Histogram = distribution en buckets + percentiles.
  • node_exporter = metriques Linux. blackbox_exporter = sondes.
  • AlertManager = routing + grouping + silencing.
  • PromQL = langage de requete (rate, sum, histogram_quantile).
  • Recording Rules = requetes pre-calculees pour performance.

4 - Architecture

0/5

Prometheus Server — les 3 composants internes : 1) Le Retrieval (scraper) interroge les targets a intervalles reguliers et recupere les metriques. 2) La TSDB (Time Series Database) stocke les donnees sur disque de maniere optimisee (compression, indexation par labels). 3) Le HTTP Server expose l'API pour les requetes PromQL, la federation et la configuration. Ces trois composants forment un seul binaire auto-suffisant.

Service Discovery : Prometheus peut decouvrir automatiquement les targets sans configuration manuelle. Modes supportes : kubernetes_sd_configs (decouvre les pods/services K8s), consul_sd_configs (services Consul), ec2_sd_configs (instances AWS), file_sd_configs (fichier JSON/YAML mis a jour dynamiquement), dns_sd_configs. En Kubernetes, c'est transparent : les pods avec les bonnes annotations sont automatiquement scrapes.

Exporters dans l'architecture : Les exporters sont des processus intermediaires qui traduisent les metriques d'un systeme tiers au format Prometheus. node_exporter ecoute sur le port 9100, blackbox_exporter sur 9115, mysqld_exporter sur 9104. Chaque exporter expose un endpoint /metrics que Prometheus vient scraper. Pour les applications modernes, on instrumente directement le code avec les librairies client (pas besoin d'exporter).

AlertManager dans l'architecture : Prometheus evalue les alerting rules regulierement. Quand une condition est remplie (ex: CPU > 90% pendant 5 min), Prometheus envoie l'alerte a AlertManager. Celui-ci deduplication, regroupe (grouping), route vers le bon receiver (Slack, PagerDuty, email) et gere les silences et inhibitions. AlertManager peut etre deploye en mode HA (haute disponibilite) avec clustering.

Pushgateway : Exception au modele pull : le Pushgateway permet aux jobs ephemeres (batch jobs, cron tasks) de POUSSER leurs metriques. Le job pousse vers le Pushgateway, et Prometheus scrape le Pushgateway comme une target classique. A utiliser UNIQUEMENT pour les jobs courts — jamais pour les services permanents (sinon on perd la detection de pannes).

Architecture Grafana : Grafana se connecte a Prometheus comme "data source" via l'API HTTP de Prometheus. Les dashboards sont composes de panels qui executent des requetes PromQL. Grafana peut aussi se connecter a Loki (logs), Tempo (traces), InfluxDB, Elasticsearch, etc. L'alerting Grafana permet de definir des alertes visuelles directement dans les panels, avec notification multi-canal.

  • Prometheus = Retrieval + TSDB + HTTP Server (un seul binaire).
  • Service Discovery = detection auto des targets (K8s, Consul, AWS).
  • Pushgateway = uniquement pour jobs ephemeres.
  • AlertManager = deduplication + routing + grouping + silencing.
  • Grafana = multi-data-sources, panels, variables, alerting.

5 - Installation

0/5

Installation Docker (la plus simple) : Prometheus : docker run -d -p 9090:9090 -v prometheus.yml:/etc/prometheus/prometheus.yml prom/prometheus. Grafana : docker run -d -p 3000:3000 grafana/grafana-oss. Prometheus ecoute sur le port 9090, Grafana sur le port 3000 (login par defaut : admin/admin). En Docker, on monte le fichier de configuration prometheus.yml en volume.

Installation avec Docker Compose : Docker Compose permet de lancer toute la stack en une commande (docker-compose up -d). On definit les services prometheus, grafana, node_exporter, alertmanager dans un fichier docker-compose.yml. Les volumes persistent les donnees. Les reseaux Docker permettent aux containers de communiquer par nom de service (ex: http://prometheus:9090).

Helm Chart sur Kubernetes : Pour un deploiement Kubernetes, on utilise Helm (gestionnaire de packages K8s). La commande : helm install prometheus prometheus-community/prometheus. Cela deploie Prometheus, AlertManager, node-exporter et kube-state-metrics automatiquement. Les valeurs par defaut sont adaptees a la plupart des clusters.

kube-prometheus-stack (recommande pour K8s) : Le chart Helm "kube-prometheus-stack" installe TOUT : Prometheus Operator, Grafana (avec dashboards pre-configures), AlertManager, node-exporter, kube-state-metrics, et des regles d'alerte par defaut. C'est la methode recommandee en production Kubernetes. Installation : helm install monitoring prometheus-community/kube-prometheus-stack.

Configuration prometheus.yml : Le fichier principal de configuration. Sections cles : global (scrape_interval, evaluation_interval), alerting (lien vers AlertManager), rule_files (fichiers de regles d'alerte et recording rules), scrape_configs (liste des jobs/targets a scraper). Chaque job definit un nom, les targets, et optionnellement des relabel_configs pour transformer les labels.

Configuration data source Grafana : Dans Grafana, on ajoute Prometheus comme source de donnees via Configuration > Data Sources > Add > Prometheus. L'URL est http://prometheus:9090 (en Docker/K8s) ou http://localhost:9090. On peut ensuite importer des dashboards pre-faits depuis grafana.com/grafana/dashboards (ex: dashboard ID 1860 pour node_exporter).

  • Prometheus = port 9090. Grafana = port 3000.
  • docker-compose = stack complete en une commande.
  • kube-prometheus-stack = methode recommandee K8s.
  • prometheus.yml = scrape_configs + rule_files + alerting.
  • Grafana login par defaut : admin/admin.

6 - Demo / Projet fil rouge

0/5

Monitorer une application : Pour monitorer une application, on l'instrumente avec une librairie client Prometheus (prometheus_client en Python, prom-client en Node.js, micrometer en Java Spring). L'application expose alors un endpoint /metrics avec des metriques custom : nombre de requetes, latence, erreurs, metriques metier (commandes traitees, utilisateurs connectes). Prometheus scrape cet endpoint et les donnees deviennent queryables en PromQL.

Creer des dashboards Grafana : Un bon dashboard suit le principe RED (Rate, Errors, Duration) pour les services et USE (Utilization, Saturation, Errors) pour les ressources. On cree des panels avec des requetes PromQL : rate(http_requests_total[5m]) pour le trafic, rate(http_requests_total{status=~"5.."}[5m]) pour les erreurs, histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[5m])) pour la latence p99. Les variables ($namespace, $instance) rendent le dashboard reutilisable.

Configurer les alertes : Les alerting rules se definissent dans un fichier YAML. Exemple : alert: HighErrorRate, expr: rate(http_requests_total{status=~"5.."}[5m]) > 0.05, for: 5m (attend 5 min avant de tirer), labels: severity: critical, annotations: description. AlertManager route l'alerte vers Slack/PagerDuty selon la severite. Toujours definir un "for" pour eviter les faux positifs.

Integration avec Jenkins : Jenkins expose des metriques via le plugin Prometheus Metrics. On peut alors monitorer : nombre de builds, duree des builds, taille de la queue, statut des jobs. Un dashboard Grafana dedie permet de visualiser la sante du CI/CD. Alerte possible si la queue depasse un seuil ou si le taux d'echec augmente.

Integration avec Kubernetes : En K8s, kube-state-metrics expose l'etat des objets (pods, deployments, nodes). node_exporter expose les metriques hardware de chaque noeud. cadvisor (integre a kubelet) expose les metriques des conteneurs (CPU, memoire par pod). Les dashboards standards couvrent : vue cluster, vue namespace, vue pod, vue node. Les alertes typiques : pod en CrashLoopBackOff, noeud NotReady, PVC presque plein.

  • Instrumentation = librairie client dans l'app (/metrics).
  • RED = Rate, Errors, Duration (services).
  • USE = Utilization, Saturation, Errors (ressources).
  • "for: 5m" dans les alertes = eviter faux positifs.
  • kube-state-metrics + node_exporter + cadvisor = monitoring K8s complet.

7 - References et bonnes pratiques

0/5

Documentation officielle : La documentation Prometheus (prometheus.io/docs) est la reference absolue. Elle couvre la configuration, PromQL, les bonnes pratiques de nommage des metriques, et les guides d'integration. La documentation Grafana (grafana.com/docs) explique la creation de dashboards, les plugins, et l'administration. Ces deux ressources sont maintenues a jour par les mainteneurs officiels.

Conventions de nommage des metriques : Les metriques doivent suivre un format strict : snake_case, prefixe par le nom du domaine (http_, node_, process_), suffixe par l'unite (_seconds, _bytes, _total pour les counters). Exemple correct : http_request_duration_seconds. Exemple incorrect : HttpRequestDuration. Les labels doivent etre de faible cardinalite (pas d'ID utilisateur dans un label, sinon explosion des series).

Bonnes pratiques d'alerting : 1) Alerter sur les SYMPTOMES, pas les causes (alerter "latence haute" plutot que "CPU haute" car la CPU haute n'impacte pas toujours les utilisateurs). 2) Chaque alerte doit avoir un runbook (documentation de resolution). 3) Eviter l'alert fatigue : trop d'alertes = on les ignore toutes. 4) Utiliser des severites : critical (appel immediat), warning (a traiter dans la journee), info (informatif).

Retention et stockage : Par defaut, Prometheus conserve les donnees 15 jours (--storage.tsdb.retention.time=15d). Pour une retention longue, utiliser Thanos ou Cortex qui ajoutent un stockage objet (S3, GCS) en backend. Victoria Metrics est une alternative performante compatible PromQL avec meilleure compression. Toujours dimensionner le disque : environ 1-2 octets par echantillon.

Scalabilite et haute disponibilite : Prometheus n'est PAS nativement clusterise. Pour la HA : lancer 2 instances identiques (meme config) — AlertManager deduplique les alertes. Pour la scalabilite : federation (un Prometheus global scrape des Prometheus locaux) ou Thanos/Cortex pour une vue unifiee multi-cluster. Grafana supporte nativement la HA avec une base de donnees partagee (PostgreSQL/MySQL).

  • Metriques en snake_case avec unite en suffixe (_seconds, _bytes, _total).
  • Alerter sur les symptomes, pas les causes.
  • Retention par defaut = 15 jours. Thanos/Cortex pour plus.
  • HA Prometheus = 2 instances identiques + AlertManager deduplication.
  • Labels = faible cardinalite obligatoire.

5. Optimisation et performance

  • Recording Rules : pre-calculer les requetes lourdes utilisees dans les dashboards.
  • Cardinalite : surveiller avec prometheus_tsdb_head_series. Au-dela de 1M series = probleme.
  • Retention : ajuster --storage.tsdb.retention.time selon l'espace disque disponible.
  • Remote Write : envoyer les metriques vers un stockage longue duree (Thanos, Mimir, Victoria Metrics).

6. Metriques Golden Signals (Google SRE)

  • Latency : temps de reponse des requetes (histogram_quantile).
  • Traffic : volume de requetes (rate sur counter).
  • Errors : taux d'erreurs (ratio 5xx / total).
  • Saturation : charge du systeme (CPU, memoire, queue).
