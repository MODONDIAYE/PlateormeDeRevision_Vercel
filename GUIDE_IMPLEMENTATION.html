2.1 - Ouvrir le fichier Jenkins

✅ Ouvrir Jenkins_Revision_Plateforme_Modou_NDIAYE.html dans VS Code
✅ Utiliser Ctrl+F pour chercher : <div class="qcm-question" data-page="p1">

2.2 - Pour chaque page (p1, p2, ... p17) :

2.2.1 - Localiser les 6 questions existantes
Chercher : <div class="qcm-question" data-page="p1"> (pour page 1)
Vous trouverez 6 blocs HTML (Q1-Q6) identiques dans structure.
2.2.2 - Copier le dernier bloc (Q6)
Localiser la 6ème question. C'est la dernière avant </div> qui ferme le qcm-grid.
2.2.3 - Coller 9 fois, modifier les IDs
Exemple pour Q7 :
<div class="qcm-question" data-page="p1"> <p class="q-text">7) Quel est le temps d'integration ideal...</p> <div class="qcm-options"> <div class="qcm-option"> <input type="radio" name="q1-7" value="a" id="q1-7a"> ... </div> </div> <button class="btn-validate" onclick="checkAnswer('q1-7','c',this,'p1')"> ... </div>
Points critiques à modifier :
  • name="q1-7" (numéro question)
  • id="q1-7a", id="q1-7b", etc.
  • checkAnswer('q1-7','c',this,'p1') (bonne réponse)
  • Le texte question doit être différent (Q7 ≠ Q6)
2.2.4 - Ajuster le score affichage
Chercher dans la page p1 :
<span class="qcm-score-display" id="qcm-score-p1">Score: 0/6</span>
Remplacer par :
<span class="qcm-score-display" id="qcm-score-p1">Score: 0/15</span>
2.2.5 - Ajouter les 9 justifications au bloc JS
À la fin du fichier, chercher :
const justifications = { 'q1-1': "CI signifie...", 'q1-2': "Sans CI...", ... };
Ajouter après la dernière justification (avant le } final) :
'q1-7': "Le temps d'integration complet doit etre < 10 minutes...", 'q1-8': "Sans CI (et sans integration frequente), les branches divergent...", ... 'q1-15': "Le choix GitHub Actions vs Jenkins depend principalement..."
2.2.6 - Enrichir les Concepts
Localiser dans la page :
<div class="concepts"> <p>...paragraphes existants...</p> </div>
Ajouter 2-3 paragraphes supplémentaires AVANT la fermeture </div>:
<p><strong>Metriques de sante CI — tableau de bord :</strong> Pour savoir si votre CI fonctionne, surveillez 3 KPIs...</p> <p><strong>Les 3 anti-patterns CI a absolument eviter :</strong> 1) "Ignorer red builds"...</p> <p><strong>L'equation simple : ROI de CI</strong> Temps economise = ...</p>
2.2.7 - Valider dans le navigateur
✅ Ouvrir Live Server (VS Code) ou ouvrir le fichier dans Firefox/Chrome
✅ Cliquer sur la page (ex: "1. CI/CD Intro")
✅ Aller au tab "QCM"
✅ Vérifier que le score affiche 0/15 (pas 0/6)
✅ Tester une réponse pour vérifier : console → pas d'erreur JS

✅ CHECKLIST — Qualité des questions

Pour chaque question ajoutée (Q7-Q15) :

  • Question est pertinente au sujet (pas de générique)
  • 4 options A/B/C/D disponibles
  • UNE SEULE bonne réponse
  • Les autres options sont plausibles (piègent si mauvaise compréhension)
  • Justification couvre pourquoi bonne réponse (2-3 phrases)
  • Justification explique le piège de la mauvaise réponse
  • Question teste concept ou décision (pas juste "facts")
  • Réponse n'est pas évidente pour débutant (force réflexion)
  • IDs HTML uniques (q{page}-{num})
  • onclick checkAnswer correct (bonne réponse + page)

Exemples de BONNES questions :

Q7 (Concept avancé) : "Quel est le temps d'integration ideal..." (force apprentissage KPI)
Q8 (Cas pratique) : "Vous avez 10 pods cassés..." (scenario real-world)
Q9 (Piège) : "Lequel est un anti-pattern..." (teste connaissance des risques)
Q14 (Scenario) : "Production issue + immediate action?" (decision-making)

Exemples de MAUVAISES questions :

❌ Q : "Quand Kubernetes a-t-il été créé?" (fact, pas concept)
❌ Q : "Quel est le numéro de version actuel?" (date rapidement)
❌ Q : "A, B, ou C?" (pas de structure, ambigü)
❌ Q : 3 réponses correctes possibles (confus, mauvais QCM)

⚠️ Pièges courants & Solutions

Piège 1 : ID question mal formé

❌ FAUX : name="q17" (oubli page number)
✅ BON : name="q1-7" (page + numéro)

Piège 2 : Oublier mise à jour du score

❌ FAUX : Score reste "0/6"
✅ BON : Augmenter à "0/15" pour chaque page

Piège 3 : HTML mal fermé

❌ Copier Q6 mais oublier fermer les balises</div>
✅ Solution : Utiliser VS Code "Beautify" (Alt+Shift+F) pour formater le code

Piège 4 : JavaScript erreur en console

Si une question ne valide pas → ouvrir DevTools (F12) → Console → chercher erreur
Causes courantes :
  • ID question manquante dans justifications
  • Guillemet mal échappé
  • Virgule manquante entre justifications

Piège 5 : Questions trop faciles

Les questions DOIVENT demander de la réflexion. Si la réponse est "évidente", c'est un échec de QCM.
Astuce : demander à un collègue de tester — si réponse en < 5 sec sans lire, question trop facile.

💡 Astuces avancées

1. Utiliser "Find and Replace" pour gain de temps

Ctrl+H → "Find and Replace" Exemple : Changer q1-6 → q1-7 (et variantes a/b/c/d) Chercher : q1-6([a-d]?) Remplacer : q1-7$1 Clic "Replace All" (avec regex activée)

2. Template HTML réutilisable (snippets VS Code)

Créer un snippet pour les questions QCM rapide :
File → Preferences → Configure User Snippets → html.json Ajouter : { "QCM Question": { "prefix": "qcm", "body": [ "<div class=\"qcm-question\" data-page=\"p$1\"><p class=\"q-text\">$2) $3</p><div class=\"qcm-options\">$0</div>" ] } }

3. Valider HTML en ligne

Utiliser un validateur HTML : https://validator.w3.org/
Copier le contenu, vérifier pas d'erreurs.

4. Tester en parallèle (plusieurs fichiers)

VS Code : Split screen (Ctrl+Alt+2) → moitié gauche = exemple, moitié droite = fichier à enrichir
Plus facile de référencer le pattern.
