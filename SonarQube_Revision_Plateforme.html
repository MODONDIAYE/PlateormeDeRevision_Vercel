1 - Problematique : Pourquoi la qualite du code ?

0/6

Le probleme du code non analyse : Sans outil d'analyse, les bugs se cachent dans le code pendant des semaines ou des mois. Les developpeurs accumulent du code mal ecrit (copier-coller, fonctions trop longues, variables mal nommees) sans s'en rendre compte. Quand un bug critique apparait en production, il est 10 a 100 fois plus cher a corriger que s'il avait ete detecte tot.

La dette technique : C'est une metaphore financiere. Comme une dette bancaire, le code de mauvaise qualite genere des "interets" : chaque modification devient plus lente, plus risquee, plus couteuse. Si on ne "rembourse" pas (refactoring), la dette s'accumule jusqu'a paralyser le projet. SonarQube mesure cette dette en jours/heures de travail necessaires.

La securite negligee : Un code non analyse peut contenir des failles de securite (injection SQL, XSS, mots de passe en dur). Ces vulnerabilites sont exploitables par des attaquants. L'analyse statique detecte ces failles AVANT que le code n'arrive en production.

L'analyse statique : C'est l'examen automatique du code source SANS l'executer. Contrairement aux tests unitaires qui verifient le comportement, l'analyse statique verifie la structure, la syntaxe, les patterns dangereux et le respect des bonnes pratiques. SonarQube est l'outil de reference pour l'analyse statique.

Qualite continue : Tout comme le CI/CD automatise les builds et les deployements, SonarQube automatise le controle qualite. A chaque commit, le code est analyse et l'equipe recoit un feedback immediat. C'est le concept de "Continuous Inspection".

  • Bug en prod = 10-100x plus cher qu'en dev. Detection precoce essentielle.
  • Dette technique = temps accumule pour corriger le code mal ecrit.
  • Analyse statique = examiner le code SANS l'executer.
  • Continuous Inspection = analyse a chaque commit.
  • Securite : failles detectees avant la production.
  • SonarQube = outil de reference pour l'analyse statique.

2 - Presentation de SonarQube

0/6

Definition : SonarQube est une plateforme open source d'analyse continue de la qualite du code. Elle detecte automatiquement les bugs, vulnerabilites, code smells et duplications dans plus de 30 langages de programmation. C'est l'outil le plus utilise au monde pour l'analyse statique en entreprise.

Createur : SonarQube est developpe par SonarSource, une entreprise suisse fondee en 2008. Le projet etait initialement appele "Sonar" avant d'etre renomme SonarQube. SonarSource propose aussi SonarCloud (version SaaS hebergee) et SonarLint (plugin IDE pour analyse en temps reel).

Editions : Community Edition (gratuite, open source, 15+ langages), Developer Edition (branches, PR decoration), Enterprise Edition (portfolio, governance), Data Center Edition (haute disponibilite, clustering).

Langages supportes : Java, JavaScript, TypeScript, Python, C#, C/C++, Go, PHP, Ruby, Kotlin, Swift, HTML, CSS, XML, et bien d'autres (30+ au total). Chaque langage possede ses propres regles d'analyse specifiques.

Ecosysteme SonarSource : SonarQube (serveur, analyse centralisee), SonarCloud (SaaS, zero maintenance), SonarLint (IDE, feedback instantane). Les trois outils partagent les memes regles d'analyse pour une experience coherente.

  • SonarSource (Suisse, 2008) = createur.
  • 30+ langages supportes.
  • 4 editions : Community, Developer, Enterprise, Data Center.
  • Community = gratuit et open source.
  • Ecosysteme : SonarQube + SonarCloud + SonarLint.
  • Outil de reference en entreprise pour l'analyse statique.

3 - Concepts Cles de SonarQube

0/6

Bug : Un bug est une erreur dans le code qui va provoquer un comportement inattendu ou un crash a l'execution. Exemple : diviser par zero, acceder a un objet null, boucle infinie. Le bug CASSERA le programme tot ou tard.

Code Smell : Un code smell est du code qui fonctionne correctement mais qui est mal ecrit. Il est difficile a lire, a maintenir ou a faire evoluer. Exemples : fonctions trop longues (>20 lignes), variables nommees "x" ou "temp", copier-coller. Ca ne casse rien aujourd'hui, mais ca ralentit l'equipe demain.

Vulnerability : Une vulnerabilite est une faille de securite dans le code qui peut etre exploitee par un attaquant. Exemples : injection SQL, mot de passe en dur dans le code, donnees sensibles non chiffrees. C'est une porte ouverte pour les hackers.

Security Hotspot : Un hotspot de securite est un morceau de code qui necessite une revue manuelle par un humain pour determiner s'il represente un risque de securite. Contrairement a une vulnerabilite (certaine), le hotspot est un "peut-etre" qui demande une verification.

Technical Debt : La dette technique est le temps total estime pour corriger TOUS les code smells du projet. SonarQube l'exprime en jours/heures (ex: "3j 4h de dette"). C'est le "prix" a payer pour remettre le code au propre.

Quality Profile : Un Quality Profile est un ensemble de regles d'analyse appliquees a un langage. Exemple : le profil "Sonar Way" pour Java contient ~500 regles. On peut personnaliser les profils pour activer/desactiver des regles selon le contexte du projet.

Quality Gate : Un Quality Gate est un ensemble de conditions pass/fail. Si le code ne respecte pas les seuils (ex: couverture > 80%, 0 bugs critiques), le gate echoue et le code est rejete. C'est le "gardien" de la qualite.

Rule, Metric, Issue, New Code : Une Rule est une verification individuelle (ex: "pas de null pointer"). Une Metric est une mesure (couverture, duplications, complexite). Un Issue est tout probleme detecte (bug, smell, vulnerability). New Code = code ajoute depuis la derniere version, c'est la zone de focus prioritaire.

  • Bug = erreur qui causera un crash. Priorite maximale.
  • Code Smell = ca marche mais c'est mal ecrit. Maintenabilite.
  • Vulnerability = faille exploitable par un attaquant.
  • Security Hotspot = code a verifier manuellement.
  • Quality Gate = conditions pass/fail (gardien).
  • New Code = focus sur le code recent (Clean as You Code).

4 - Architecture de SonarQube

0/6

Scanner (Analyseur) : C'est le composant qui s'execute sur la machine du developpeur ou dans le CI/CD. Il lit le code source, applique les regles d'analyse et envoie les resultats au serveur SonarQube. Il ne stocke rien lui-meme. Plusieurs scanners existent : SonarScanner CLI, Scanner for Maven, Scanner for Gradle, Scanner for .NET.

Serveur Web : C'est l'interface utilisateur de SonarQube (port 9000 par defaut). Il affiche les tableaux de bord, les issues, les metriques, et permet de configurer les projets, les Quality Profiles et les Quality Gates. Il expose aussi une API REST.

Compute Engine (CE) : C'est le moteur de calcul en arriere-plan. Quand un scanner envoie un rapport, le Compute Engine le traite : il calcule les metriques, applique les Quality Gates, detecte les nouveaux issues vs les anciens. Un seul rapport est traite a la fois (file d'attente).

Base de donnees : SonarQube necessite une base de donnees externe pour stocker les projets, issues, metriques et configurations. PostgreSQL est recommande en production. Les autres options sont Microsoft SQL Server et Oracle. La base embarquee H2 est UNIQUEMENT pour les tests.

Elasticsearch : SonarQube utilise un moteur Elasticsearch integre pour indexer les issues et permettre la recherche rapide dans l'interface. Il est embarque dans le serveur SonarQube (pas besoin de l'installer separement).

  • Scanner = analyse le code, envoie le rapport.
  • Web Server = interface UI + API REST (port 9000).
  • Compute Engine = traite les rapports un par un.
  • PostgreSQL recommande en production.
  • H2 = tests UNIQUEMENT, jamais en prod.
  • Elasticsearch integre pour la recherche.

5 - Installation de SonarQube

0/6

Prerequis systeme : Java 17 est requis (SonarQube est ecrit en Java). PostgreSQL 13+ est recommande. Minimum 2 Go de RAM pour le serveur, 1 Go pour Elasticsearch. Le systeme doit permettre au moins 65536 fichiers ouverts (vm.max_map_count=524288 sous Linux).

Installation Docker (recommandee) : La methode la plus simple. Commande : docker run -d --name sonarqube -p 9000:9000 sonarqube:lts-community. Pour la production avec PostgreSQL : utiliser docker-compose avec un volume pour les donnees et un conteneur PostgreSQL separe.

Installation manuelle : Telecharger l'archive ZIP depuis sonarqube.org, extraire, configurer sonar.properties (connexion base de donnees), puis lancer bin/[OS]/sonar.sh start (Linux) ou StartSonar.bat (Windows). Premier acces : admin/admin (changer immediatement).

SonarScanner CLI : Apres le serveur, il faut installer le scanner pour analyser les projets. Telecharger sonar-scanner, ajouter au PATH. Creer un fichier sonar-project.properties a la racine du projet avec : sonar.projectKey, sonar.sources, sonar.host.url, sonar.token.

Token d'authentification : Pour se connecter au serveur, le scanner utilise un token genere dans l'interface (Mon Compte > Securite > Generer Token). Ce token remplace le login/mot de passe et est plus securise. Ne JAMAIS le mettre en clair dans le code.

  • Java 17 requis, PostgreSQL 13+ recommande.
  • Docker = methode la plus simple.
  • Port 9000 par defaut.
  • Premier login : admin/admin (changer immediatement).
  • Token > login/mot de passe pour le scanner.
  • sonar-project.properties = config du projet.

6 - Integration CI/CD

0/6

Integration Jenkins : SonarQube s'integre dans un pipeline Jenkins via le plugin "SonarQube Scanner for Jenkins". Dans le Jenkinsfile, on utilise le bloc withSonarQubeEnv('SonarQube') { sh 'sonar-scanner' } pour lancer l'analyse. Le plugin injecte automatiquement l'URL et le token.

Quality Gate dans le pipeline : Apres l'analyse, on peut attendre le resultat du Quality Gate avec waitForQualityGate(). Si le gate echoue (ex: trop de bugs), le pipeline Jenkins echoue aussi. Cela empeche le deploiement de code de mauvaise qualite.

Maven/Gradle : Pour les projets Java, pas besoin du scanner CLI. Maven : mvn sonar:sonar -Dsonar.token=MON_TOKEN. Gradle : ./gradlew sonarqube -Dsonar.token=MON_TOKEN. Le plugin analyse automatiquement les sources et les tests.

PR Decoration (Pull Request) : Avec l'edition Developer+, SonarQube peut commenter directement les Pull Requests sur GitHub/GitLab/Azure DevOps. Les developpeurs voient les issues directement dans leur PR, sans quitter leur outil de travail.

Branch Analysis : L'edition Developer+ permet d'analyser chaque branche separement. La branche main est la reference, et les feature branches sont comparees a elle. Cela permet de detecter les regressions avant le merge.

  • withSonarQubeEnv() = bloc Jenkins pour l'analyse.
  • waitForQualityGate() = bloquer si qualite insuffisante.
  • Maven : mvn sonar:sonar.
  • PR Decoration = commentaires dans les PR.
  • Branch Analysis = comparer a main.
  • Quality Gate echoue = pipeline echoue.

7 - References et Bonnes Pratiques

0/5

Clean as You Code : C'est la philosophie officielle de SonarQube. Au lieu d'essayer de corriger toute la dette technique d'un coup (impossible), on se concentre sur le New Code : tout nouveau code doit passer le Quality Gate. Progressivement, le projet s'ameliore.

Sonar Way : C'est le Quality Profile par defaut recommande par SonarSource. Il contient les regles les plus importantes et est mis a jour automatiquement. Pour la plupart des projets, c'est un excellent point de depart sans aucune configuration.

Metriques essentielles : Coverage (couverture de tests, objectif > 80%), Duplications (code duplique, objectif < 3%), Complexity (complexite cyclomatique, fonctions simples), Reliability Rating (A a E selon les bugs), Security Rating (A a E selon les vulnerabilites).

API REST SonarQube : SonarQube expose une API complete sur /api/. Exemples : GET /api/measures/component (metriques), GET /api/issues/search (issues), POST /api/qualitygates/project_status (statut gate). Documentation integree sur localhost:9000/web_api.

Bonnes pratiques : 1) Toujours utiliser un Quality Gate. 2) Analyser a chaque commit (CI). 3) Ne jamais ignorer les Security Hotspots. 4) Former l'equipe aux concepts SonarQube. 5) Commencer avec Sonar Way puis personnaliser. 6) Monitorer la dette technique chaque sprint.

  • Clean as You Code = focus sur le New Code.
  • Sonar Way = profil par defaut recommande.
  • Coverage > 80%, Duplications < 3%.
  • API REST sur /api/ (documentation integree).
  • Analyser a chaque commit, pas une fois par semaine.

Severity des Issues

  • BLOCKER - Bug avec forte probabilite d'impact en production
  • CRITICAL - Bug avec faible probabilite OU faille de securite
  • MAJOR - Defaut qualite susceptible d'impacter la productivite
  • MINOR - Defaut qualite avec faible impact
  • INFO - Ni bug ni defaut, juste une remarque

Ratings (A a E)

  • A = 0 bugs/vulnerabilites (excellent)
  • B = au moins 1 Minor
  • C = au moins 1 Major
  • D = au moins 1 Critical
  • E = au moins 1 Blocker
